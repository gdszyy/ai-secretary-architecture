## 1. 权限对比清单

## 2. 关键发现与差距分析

### 2.1 核心敏感权限已具备
经过测试，该机器人已经具备了 `im:message.group_msg` 这一最核心的敏感权限。在测试脚本中，我们成功调用了 `/open-apis/im/v1/messages` 接口，拉取到了机器人所在群（chat_id: `oc_336b676a6d19c07bcc3b54854bce816e`）的历史消息。
这意味着，**无需再次走漫长的管理员审批流程**，该应用已经可以直接用于全量监听群聊，这是项目实施中的一个重大利好。

### 2.2 飞书妙记 API 接口不可用
在测试 `minutes:minute:readonly` 权限时，无论是调用 `open.larksuite.com` 还是 `open.feishu.cn` 域名的妙记 API（`/open-apis/minutes/v1/minutes`），服务端均返回了 `404 page not found` 错误。
这表明，该租户或应用所处的飞书环境**暂未开放官方妙记 API**，或者该端点已被飞书官方下线/更改。

## 3. 后续行动建议

基于上述分析，我们对实施方案提出以下建议：

1. **复用当前应用（强烈推荐）**
   鉴于 `cli_a9d985cd40f89e1a` 已经具备了最难申请的 `im:message.group_msg` 全量群消息监听权限，强烈建议直接复用该应用作为“AI 项目秘书”的载体。后端开发人员只需在飞书开发者后台配置 Webhook 回调地址（订阅 `im.message.receive_v1` 事件）即可立即开始接收群消息。

2. **妙记接入降级为“方案 B”**
   由于官方妙记 API 不可用（返回 404），原定的“方案 A：定时轮询自动拉取”无法实施。建议直接启用《接入指导手册》中定义的“**方案 B：手动触发导出**”。
   在实际操作中，项目经理（PM）可以通过以下方式将会议记录推送到缓冲区：
   - 方式一：在 Lark 群聊中 `@机器人` 并附上妙记链接，由 AI 秘书通过浏览器自动化或内部接口拉取。
   - 方式二：手动将妙记导出为 `.txt` 文件，通过内部上传接口提交。
