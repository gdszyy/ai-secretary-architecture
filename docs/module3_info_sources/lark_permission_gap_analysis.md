## 2. 关键发现与分析

### 2.1 群聊机器人：所有权限就绪，可立即开始

该应用已具备包括最核心的 `im:message.group_msg` 敏感权限在内的所有群聊相关权限。测试中成功拉取到了机器人所在群的历史消息（`code=0`），验证了权限的实际有效性。

**Lark 群聊机器人接入无任何权限阻碍，可立即进入开发阶段。**

### 2.2 飞书妙记：权限有效，API 端点需使用正确路径

早期测试中，我们错误地调用了 `/open-apis/minutes/v1/minutes`（列表接口），该接口返回 404，原因是**飞书妙记 API 没有"获取全部妙记列表"的接口**，而是需要提供具体的 `minute_token` 才能查询单篇妙记。

正确的 API 端点为：

| 接口名称 | HTTP 方法 | 正确路径 |
| :--- | :--- | :--- |
| 获取妙记基础信息 | `GET` | `https://open.feishu.cn/open-apis/minutes/v1/minutes/{minute_token}` |
| 获取妙记文字记录 | `GET` | `https://open.feishu.cn/open-apis/minutes/v1/minutes/{minute_token}/transcript` |

使用正确路径（含有效 `minute_token`）调用时，返回 `code=2091002, msg=resource not found`，这是因为测试用的 `minute_token` 是占位符，**并非权限错误**（权限错误码为 `2091005, permission deny`）。这证明 `minutes:minutes:readonly` 权限已实际生效。

## 3. 接入方案调整

鉴于妙记 API 没有列表接口，接入方案需做如下调整：

**原方案（定时轮询全量妙记列表）** → 不可行，飞书无此接口。

**调整后方案**：通过以下两种方式获取 `minute_token`，再调用 API 拉取内容：

| 方式 | 操作步骤 | 自动化程度 |
| :--- | :--- | :--- |
| **方式 A：监听日历事件** | 订阅 `calendar.event.changed_v1` 事件，当会议结束时从日历事件的 `meeting_minutes_url` 字段中解析 `minute_token` | 高（全自动） |
| **方式 B：PM 主动触发** | PM 在 Lark 群聊中 @机器人 并粘贴妙记链接，机器人从 URL 中提取 `minute_token` 后自动调用 API | 中（需 PM 一次操作） |

**建议优先实现方式 B**（开发成本低，可立即落地），同时评估方式 A 的可行性作为长期优化方向。
