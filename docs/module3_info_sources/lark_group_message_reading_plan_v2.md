## 1. 架构理念：从事件驱动转向 Skill 定时任务

在团队群聊场景中，信息的产生往往是高频、跳跃且碎片化的。传统的“实时 Webhook 事件驱动”架构在处理这类信息时存在明显的缺陷：单条消息上下文极度匮乏，极易触发过度打扰的即时追问。

为此，本方案将架构重心调整为 **Skill 定时任务驱动 (Scheduled Skill-Driven)**：
通过定期（如每小时、每天下班前）运行专门的 Agent Skill，批量拉取群聊历史记录。系统将这一段时间内的群聊视为一个完整的上下文窗口，运用大模型进行全局视角的“话题拆解（Thread Separation）”，从中提取关键信息，并与项目的模块进度（Lark 看板/Meegle）进行对齐和并轨。

### 1.1 核心流水线 (Pipeline)

1.  **定时拉取 (Fetch)**: 定时任务触发 `lark-group-monitor` 技能，通过 `/im/v1/messages` 接口拉取过去 N 小时的群聊消息流。
2.  **全局拆解 (Decompose)**: LLM 将线性且跳跃的消息流，基于语义关联性拆解为多个并行的话题线程（Threads）。
3.  **并轨提取 (Extract & Merge)**: 针对每个独立的话题，提取核心要素（状态、缺陷、需求），并尝试将其与 `project_context.json` 中的业务模块（如“用户系统”、“支付网关”）对齐。
4.  **智能追问 (Inquire)**: 对于信息不足但对话显然已经结束的话题，机器人主动在群内 `@` 相关责任人，并引用上下文进行追问。
5.  **进度更新 (Sync)**: 完整度达标的信息，直接推入缓冲池（Buffer），随后更新至 Lark 看板或 Meegle。

## 2. 核心算法：多话题并行拆解与跳跃内容归并

群聊的本质是“多路复用的异步通信”。例如：A 和 B 在排查支付 Bug 的同时，C 突然插话询问活动页面的设计进度，随后 A 又回复了 C，再继续和 B 讨论 Bug。

### 2.1 基于全局视角的 LLM 拆解算法

由于我们采用了定时批处理架构，LLM 拥有了“上帝视角”，能够看到整个对话的始末。我们采用以下策略进行拆解：

1.  **上下文窗口切分**: 将拉取到的消息（如 100 条）作为一个完整的语义块输入给 LLM。
2.  **实体与提及锚定**: LLM 首先识别消息中的显式锚点：
    *   **@ 提及关系**: 明确的回复链路。
    *   **业务实体**: 模块名称（如“风控系统”）、术语、特定的 Issue ID。
3.  **跳跃归并与重叠分配**:
    *   **跳跃归并**: 即使两条消息相隔 20 条其他发言，只要语义实体一致（如都在讨论同一个 Bug 现象），LLM 必须将其归并到同一个 Thread 中。
    *   **重叠分配**: 当某条消息同时回答了两个问题时（如“支付修好了，另外活动页设计稿我也看了没问题”），该消息 ID 允许同时出现在两个 Thread 的 `messages` 列表中。

### 2.2 并轨到模块进度

拆解出独立的话题（Thread）后，系统需要将其“并轨”到实际的项目进度管理中。

*   **输入注入**: 在 LLM 拆解时，系统必须将最新的 `project_context.json`（包含模块列表、当前状态、负责人）作为 System Prompt 注入。
*   **归属判定**: LLM 在输出 `ThreadEvent` 时，必须强制输出 `module_name` 和 `impacted_status` 字段。
*   **进度映射**: 例如，LLM 识别出“支付路由修复”话题已得出结论，将其归属于“财务系统”，并生成一个 `Status Update` 的意图推入 Buffer。

## 3. 主动追问机制：基于上下文的 @人提问策略

在批量处理模式下，经常会遇到“对话已经自然结束，但关键要素依然缺失”的情况（例如：群里说“那个缓存穿透的问题先这样吧”，但没说具体怎么改、谁来改）。

### 3.1 触发追问的判定条件

系统在完成话题拆解后，会对每个 Thread 进行“完整度与闭环检测”：
1.  **对话已结束**: 距离该话题的最后一条消息已超过 30 分钟。
2.  **信息不完整**: 根据意图类型（如 `Bug Report`），缺失了关键字段（如 `assignee` 或 `solution`），导致完整度评分低于 80 分。
3.  **意图非闲聊**: 话题被识别为具体的业务意图，而非 `casual_chat`。

### 3.2 追问动作与话术生成

满足上述条件时，系统将触发主动追问：

1.  **定位责任人**: 根据该 Thread 的参与者（`participants`）以及 `project_context.json` 中对应模块的 `owner`，确定被 `@` 的人员。优先 `@` 话题的最后发言人或模块负责人。
2.  **引用上下文**: 为了让被 `@` 的人立刻回忆起上下文，机器人必须**引用（Quote）**该话题中最核心的一条消息，或者总结该话题的背景。
3.  **精确提问**: 生成的话术必须明确指出缺失了什么信息。

**话术示例 (群聊机器人发送)**:

> 引用: [A: 那个缓存穿透的问题先这样吧，明天再说。]
>
> 🤖 **@A** 监控到关于“用户系统缓存穿透”的讨论已暂停。为了同步看板进度，请问：
> 1. 这个问题的最终解决方案定下来了吗？
> 2. 预计什么时候修复上线？
> (直接回复本条消息即可)

### 3.3 追问后的闭环 (下一周期扫描)

*   为了实现完全无后端的纯粹 Skill 架构，系统不依赖 Webhook 实时接收用户的回复。
*   当用户在群内回复了机器人的追问后，该回复将静静躺在群聊记录中。
*   在 **下一个定时周期**（如下一小时），`lark-group-monitor` 再次启动并拉取新消息时，LLM 会将新拉取到的回复消息，与暂存在 Bitable `Pending Threads 表` 中的未闭环话题进行上下文合并。
*   合并后重新计算完整度，若达标，则正式推入 Buffer 并更新至看板，同时在 Bitable 中将该 Thread 标记为 `completed`。

## 5. 基础设施：后端依赖与数据存储方案

针对 Skill-Driven 定时任务架构，基础设施和数据存储方案必须适应“周期性唤醒、批处理、状态暂存”的特点。

### 5.1 是否需要后端能力？

**结论：完全无服务器化 (Serverless / Agent-Native)，零后端依赖。**

*   **定时拉取与拆解 (Fetch & Decompose)**：依赖 Agent 调度平台（如 Manus 定时任务或 GitHub Actions Cron）定期唤醒 `lark-group-monitor` Skill 执行拉取和 LLM 推理。
*   **进度推入看板 (Sync)**：Skill 执行完毕后，直接调用 Lark Bitable API 或 Meegle API 写入结果。
*   **追问闭环 (Inquire & Reply)**：系统不部署 Webhook 服务接收实时事件。机器人在群内主动 `@` 用户追问后，直接结束当前周期的运行。用户的回复将在 **下一个定时周期** 的批量拉取中被扫描到，并与 Bitable 中暂存的 Thread 上下文进行合并闭环。这种方式虽然存在一定的处理延迟（取决于定时任务的频率，如 30 分钟或 1 小时），但换取了架构的极致轻量化和免运维。

### 5.2 数据存储方案设计

在 Skill-Driven 架构下，由于是无状态的脚本定时执行，必须引入持久化存储来记录“拉取游标”和“未完成的话题状态”。

**方案一：Lark 多维表格 (Bitable) 作为轻量级数据库 (推荐)**

为了极致的“免运维”和与飞书生态的深度融合，直接使用一个隐藏的 Lark 多维表格作为数据库。

*   **表1：Cursor 表 (拉取游标)**
    *   字段：`group_id`, `last_fetched_message_id`, `last_fetched_time`
    *   作用：每次 Skill 启动时，先读取此表获取上次拉取的位置，避免重复拉取或漏拉。
*   **表2：Pending Threads 表 (待补全话题池)**
    *   字段：`thread_id`, `topic_summary`, `missing_fields`, `last_inquiry_time`, `status` (waiting_reply, completed, timeout)
    *   作用：暂存那些被判定为“信息不足”且已发起追问的话题。当用户的回复到来（或下一次 Skill 扫描到回复）时，从这里提取上下文进行合并。

**方案二：SQLite + S3 (适合自托管 Agent)**

如果使用本地运行的 Agent 框架（如 `multi-agent-hub` 技能中提到的 Git 仓库持久化或本地 SQLite）：

*   复用 `prereq_data_assessment.md` 中设计的 `processed_messages` SQLite 表进行去重。
*   新增 `pending_threads` 表存储未闭环的上下文。
*   由于 Agent 运行环境可能随时销毁，SQLite 文件需要在每次执行结束后同步到云端（如 S3 或提交到 Git 仓库的一个隐藏分支）以保持状态。

**最终决策：采用方案一（Lark Bitable）**

系统将采用纯 Lark Bitable 存储方案。这不仅实现了完全的无后端架构，免去了数据库部署和维护成本，还为项目经理提供了一个天然的“可视化”操作界面。项目经理可以随时在表格中查看系统正在追踪哪些未闭环的话题，甚至允许人工直接在表格里手动补全信息，从而绕过下一个扫描周期，立即触发看板的更新。
