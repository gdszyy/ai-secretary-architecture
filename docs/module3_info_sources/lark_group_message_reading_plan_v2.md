> **架构重构说明 (2026-04-20)**：
> 本方案已从“事事追问的待办生成器”重构为“高价值事实提取器”。
> 核心调整：废弃对常规缺陷/需求的群内自动 @ 追问；取消“未闭环话题”对调度频率的强制提升；强化群名作为上下文辅助；彻底解耦项目级事实与个人秘书级跟进。

## 1. 架构理念：从事件驱动转向 Skill 定时任务

在团队群聊场景中，信息的产生往往是高频、跳跃且碎片化的。传统的“实时 Webhook 事件驱动”架构在处理这类信息时存在明显的缺陷：单条消息上下文极度匮乏，极易触发过度打扰的即时追问。

为此，本方案将架构重心调整为 **Skill 定时任务驱动 (Scheduled Skill-Driven)**：
通过定期运行专门的 Agent Skill，批量拉取群聊历史记录。系统将这一段时间内的群聊视为一个完整的上下文窗口，运用大模型进行全局视角的“话题拆解（Thread Separation）”，从中提取关键信息，并与项目的模块进度（Lark 看板/Meegle）进行对齐和并轨。

### 1.1 群组动态优先级与冷启动探索

并非所有群组都具有同等的信息密度，如果所有群都采用固定的拉取频率（如每小时一次），会造成极大的 Token 浪费或重要信息遗漏。因此，调度机制必须是**动态且智能**的。

#### 1.1.1 冷启动探索机制 (Cold Start Exploration)

当机器人被邀请进入一个新群组时，系统对该群的性质一无所知。因此，**前 N 次（如前 3 次）的同步将被标记为“探索期”**。

在探索期内，LLM 的主要任务不是急于提取进度，而是**分析群组画像**：
*   **群组定位**：这是核心研发群、产品需求群、运维报警群，还是闲聊摸鱼群？（**群名**是极其重要的上下文辅助，如“前端组”、“数据对接组”直接决定了提取信息的领域边界）
*   **活跃时段**：消息主要集中在什么时间段？
*   **核心人物**：谁是主要的信息发布者和决策者？
*   **业务关联度**：讨论的内容与 `project_context.json` 中的哪些模块最相关？

探索期结束后，LLM 将输出一份该群的“画像报告”，并据此初始化该群的**基础调度优先级**。

#### 1.1.2 动态优先级调度 (Dynamic Scheduling)

进入平稳运行期后，系统将根据每次拉取的结果，动态评估**下一次**是否需要同步该群，以及同步的间隔。评估因子包括：

*   **近期活跃度 (Frequency)**：过去几个周期内消息数量的趋势。如果一个群连续几天无人发言，其调度间隔将自动拉长（如从每小时衰减到每天一次）。
*   **内容重要性 (Importance)**：上一次提取出的高价值事实（重大决策、里程碑、风险）数量。如果一个群虽然话多，但 LLM 判定 90% 都是闲聊或常规 Bug，其调度优先级也会被降低。
*   *(已废弃)* ~~未闭环话题 (Pending Threads)：如果该群存在已发起追问但尚未闭环的话题，其调度优先级将被强制提升。~~ （新架构下，常规任务不再追问，重大决策/风险的追问频率由全局配置控制，不再影响单群调度频率）

在 Bitable 的 `Cursor 表` 中，除了记录拉取游标外，将新增 `next_scheduled_time` 字段。每次 Skill 启动时，只拉取当前时间已经超过 `next_scheduled_time` 的群组。

### 1.2 核心流水线 (Pipeline)

1.  **定时拉取 (Fetch)**: 定时任务触发 `lark-group-monitor` 技能，通过 `/im/v1/messages` 接口拉取过去 N 小时的群聊消息流。
2.  **富媒体解析前评估 (Media Parse Evaluation)**: 针对消息中的图片、链接、文档，先通过 LLM 进行低成本评估，判断其**解析价值**和**信息获取目标**。只有评分达标的内容才会触发高成本的实际解析。
3.  **全局拆解 (Decompose)**: LLM 将带有富媒体摘要的线性且跳跃的消息流，基于语义关联性拆解为多个并行的话题线程（Threads）。
4.  **价值过滤与并轨 (Filter & Merge)**: 针对每个独立的话题，判断其价值定位（重大决策、里程碑事实、风险阻塞、常规任务、个人跟进）。仅将高价值事实与 `project_context.json` 中的业务模块对齐。
5.  **静默沉淀 (Silent Archive)**: 完整度达标的高价值信息，直接推入缓冲池（Buffer），随后静默更新至 Lark 看板或风险池。
6.  **极简追问 (Minimal Inquire)**: 仅当“重大决策”缺失关键上下文，或“风险/阻塞”需要明确责任人时，才在群内 `@` 相关人员确认一次。常规任务信息不足直接丢弃，绝不追问。

## 2. 核心算法：多话题并行拆解与富媒体解析控制

### 2.1 富媒体解析前评估机制 (Media Parse Evaluation)

在群聊中，用户经常发送大量的图片（截图、表情包）、链接（需求文档、外部新闻）和文件。如果对所有富媒体内容都进行深度解析（如调用 Vision 模型或抓取网页全文），将导致 Token 成本失控和处理时间过长。

因此，在进入全局拆解之前，必须引入一个**低成本的评估拦截层 (Evaluator)**：

1. **提取上下文线索**: 提取富媒体前后的 3-5 条文本消息，以及当前群组的业务关联度。
2. **LLM 快速评估**: 让一个小参数模型（如 gpt-4o-mini）输出三项决策：
   * **信息获取目标 (Parse Goal)**: 解析这个内容最重要要拿到什么？
   * **解析价值评分 (Relevance Score)**: 0-100 分。90分以上为架构图/关键决策文档；40分以下为闲聊表情包或无关链接。
   * **停止解析逻辑 (Stop Condition)**: 何时可以提前停止？
3. **成本控制执行**:
   * 评分低于阈值（如 < 50）的媒体直接跳过，仅保留跳过原因。
   * 评分达标的媒体，将 `Parse Goal` 和 `Stop Condition` 作为 Prompt 传递给下游的实际解析器，强制解析器聚焦提取。

### 2.2 基于全局视角的 LLM 拆解算法

群聊的本质是“多路复用的异步通信”。由于我们采用了定时批处理架构，LLM 拥有了“上帝视角”，能够看到整个对话的始末。我们采用以下策略进行拆解：

1.  **上下文窗口切分**: 将拉取到的消息（如 100 条）作为一个完整的语义块输入给 LLM。
2.  **实体与提及锚定**: LLM 首先识别消息中的显式锚点：
    *   **@ 提及关系**: 明确的回复链路。
    *   **业务实体**: 模块名称（如“风控系统”）、术语、特定的 Issue ID。
3.  **跳跃归并与重叠分配**:
    *   **跳跃归并**: 即使两条消息相隔 20 条其他发言，只要语义实体一致，LLM 必须将其归并到同一个 Thread 中。
    *   **重叠分配**: 当某条消息同时回答了两个问题时，该消息 ID 允许同时出现在两个 Thread 的 `messages` 列表中。

### 2.3 价值过滤与并轨

拆解出独立的话题（Thread）后，系统需要将其“并轨”到实际的项目进度管理中。

*   **输入注入**: 在 LLM 拆解时，系统必须将最新的 `project_context.json`（包含模块列表、当前状态、负责人）以及**群名**作为 System Prompt 注入。
*   **价值判定**: LLM 必须将 Thread 归类为新架构下的五大意图之一（重大决策、里程碑事实、风险阻塞、常规任务、个人跟进）。
*   **进度映射**: 例如，LLM 识别出“Lsport 数据源接入因牌照问题受阻”，将其归类为 `Risk/Blocker`，归属于“数据接入层”，并推入 Buffer 的风险池。

## 3. 极简追问机制：告别“待办生成器”

在旧架构中，系统试图将群聊中的每一个 Bug 和需求都追踪到闭环，导致群内充斥着机器人的 `@` 追问。新架构彻底废弃了这一模式。

### 3.1 触发追问的严苛条件

系统仅在以下**极少数**情况下触发主动追问：
1.  **意图必须是极高价值**: 仅限 `Major Decision`（重大决策）或 `Risk/Blocker`（风险/阻塞）。
2.  **缺失致命信息**: 例如，群里说“决定换掉 SR 数据源”，但没说换成哪家（缺失决策结论）；或者“牌照申请卡住了”，但没说谁去跟进（缺失风险责任人）。
3.  **对话已冷寂**: 距离该话题的最后一条消息已超过 30 分钟。

**常规缺陷/需求 (Routine Task) 绝对不触发追问。** 如果信息不足以在 Meegle 建单，直接丢弃。

### 3.2 追问动作与话术生成

满足上述严苛条件时，系统将触发一次性追问：

1.  **定位责任人**: 根据该 Thread 的参与者以及模块负责人，确定被 `@` 的人员。
2.  **引用上下文**: 机器人必须**引用（Quote）**该话题中最核心的一条消息。
3.  **精确且克制的提问**: 话术必须明确指出缺失了什么致命信息，且语气应是“为了记录事实”而非“催促进度”。

**话术示例 (群聊机器人发送)**:

> 引用: [A: SR 商务不配合，我们用新公司与 SR 对接，希望换一个商务。]
>
> 🤖 **@A** 监控到关于“更换 SR 数据源商务”的重大决策讨论。为了完善项目决策记录，请问：
> 最终决定由哪家新公司出面进行对接？
> (直接回复本条消息即可，系统将自动归档)

### 3.3 追问后的闭环 (下一周期扫描)

*   当用户在群内回复了机器人的追问后，该回复将静静躺在群聊记录中。
*   在 **下一个定时周期**，`lark-group-monitor` 再次启动并拉取新消息时，LLM 会将新拉取到的回复消息，与暂存在 Bitable `Pending Threads 表` 中的未闭环话题进行上下文合并。
*   合并后重新计算完整度，若达标，则正式推入 Buffer 并更新至看板，同时在 Bitable 中将该 Thread 标记为 `completed`。

## 5. 基础设施：后端依赖与数据存储方案

针对 Skill-Driven 定时任务架构，基础设施和数据存储方案必须适应“周期性唤醒、批处理、状态暂存”的特点。

### 5.1 是否需要后端能力？

**结论：完全无服务器化 (Serverless / Agent-Native)，零后端依赖。**

*   **定时拉取与拆解 (Fetch & Decompose)**：依赖 Agent 调度平台定期唤醒 `lark-group-monitor` Skill 执行拉取和 LLM 推理。
*   **进度推入看板 (Sync)**：Skill 执行完毕后，直接调用 Lark Bitable API 或 Meegle API 写入结果。
*   **追问闭环 (Inquire & Reply)**：系统不部署 Webhook 服务接收实时事件。机器人在群内主动 `@` 用户追问后，直接结束当前周期的运行。用户的回复将在 **下一个定时周期** 的批量拉取中被扫描到，并与 Bitable 中暂存的 Thread 上下文进行合并闭环。

### 5.2 数据存储方案设计

在 Skill-Driven 架构下，由于是无状态的脚本定时执行，必须引入持久化存储来记录“拉取游标”和“未完成的话题状态”。

**方案一：Lark 多维表格 (Bitable) 作为轻量级数据库 (推荐)**

为了极致的“免运维”和与飞书生态的深度融合，直接使用一个隐藏的 Lark 多维表格作为数据库。

*   **表1：Cursor 表 (拉取游标与动态调度)**
    *   字段：`group_id`, `last_fetched_message_id`, `last_fetched_time`, `next_scheduled_time`, `priority_score`
    *   作用：记录上次拉取位置避免重复。同时，根据上一次 LLM 对群组重要性和频率的评估结果，更新 `next_scheduled_time`。每次 Skill 唤醒时，只处理已到期的群组。
*   **表2：Pending Threads 表 (待补全话题池)**
    *   字段：`thread_id`, `topic_summary`, `missing_fields`, `last_inquiry_time`, `status` (waiting_reply, completed, timeout)
    *   作用：暂存那些被判定为“信息不足”且已发起追问的**高价值话题**。当用户的回复到来（或下一次 Skill 扫描到回复）时，从这里提取上下文进行合并。

**最终决策：采用方案一（Lark Bitable）**

系统将采用纯 Lark Bitable 存储方案。这不仅实现了完全的无后端架构，免去了数据库部署和维护成本，还为项目经理提供了一个天然的“可视化”操作界面。项目经理可以随时在表格中查看系统正在追踪哪些未闭环的重大决策/风险，甚至允许人工直接在表格里手动补全信息，从而绕过下一个扫描周期，立即触发看板的更新。
