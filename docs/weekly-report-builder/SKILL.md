# 周报网页构建器 (Weekly Report Builder)

本技能指导如何通过三个交互式步骤，将用户提供的原始周报文档或数据，逐步转化为结构化的 React/TypeScript 周报网页呈现。

**核心原则：** 这是一个**交互式**的构建过程。在每个阶段结束时，必须向用户展示当前阶段的成果，并等待用户确认或补充后，才能进入下一阶段。

## 阶段 1：解析与拆解 (Analysis & Breakdown)

**目标**：解析输入文档，提取核心业务逻辑，拆解业务模块，并发现框架外洞察。

1. **读取分析框架**：首先读取 `references/analysis_framework.md`，了解预设的 5 大业务模块及评估模式。
2. **执行解析**：运行 `scripts/analysis_engine.py` 对用户提供的文档进行深度解析，提取核心业务定位、商业模式闭环和北极星指标，并将内容映射到预设的业务模块中。
3. **洞察与异常检测**：运行 `scripts/insight_detector.py`。如果发现框架外的新模式、反常识数据或隐性关联，必须单独列出并暂停流程，等待用户决策。
4. **用户交互 (必须执行)**：
   - 运行 `scripts/phase1_interaction.py` 管理交互状态。
   - 向用户展示解析结果，要求用户结合提炼的内容进行补充，完善商业逻辑和业务架构。
   - **特别提醒**：要求用户额外确认里程碑模块的准确性和当前状态。
   - 等待用户回复。

## 阶段 2：模块提炼与优先级划分 (Module Extraction & Prioritization)

**目标**：根据第一阶段确认的业务架构，提炼可用的周报呈现模块，并划分优先级。

1. **读取组件规范**：在用户确认第一阶段结果后，读取 `references/component_library.md`，了解首屏模块和其他模块的定义。
2. **提炼模块**：运行 `scripts/module_extractor.py`，提炼首屏模块 (P0) 和其他模块 (P1/P2)。
   - **数据完整性评估**：在推荐模块时，必须评估数据信息来源的完整性。如果发现可绘制成图表的数据类模块（如漏斗转化、指标趋势矩阵、用户分层气泡图等）较少或完全没有，**必须主动建议用户上传更结构化的数据（如 Excel/CSV 文件）**，以丰富周报的可视化呈现。
3. **设计导航**：运行 `scripts/navigation_designer.py`，设计从首屏模块下钻 (Drill-down) 或通过金刚区跳转到其他模块的逻辑。
4. **用户交互 (必须执行)**：
   - 运行 `scripts/phase2_interaction.py` 管理交互状态。
   - 向用户展示提炼出的呈现模块列表及优先级划分，解释每个模块的呈现意图。
   - **如果触发了数据完整性评估的预警，在此步骤必须向用户提出上传数据的建议。**
   - 要求用户选定最终需要在网页中呈现的模块。
   - 如果用户调整优先级或模块列表，重新触发优先级评分，循环至用户确认。
   - 等待用户回复。

## 阶段 3：组件选型与数据绑定 (Component Selection & Data Binding)

**目标**：为用户选定的模块匹配具体的 UI 组件，明确数据需求和交互逻辑，最终生成网页代码。

1. **匹配组件**：运行 `scripts/component_recommender.py`，为每个选定的模块推荐合适的 UI 组件。如果推荐结果中没有合适的预设组件，或需要创建预设组件库以外的新组件，**必须先读取 `references/hero_component_design_principles.md`**（首屏组件）或直接参考 `references/component_brainstorm.md`（所有组件的设计方案），再进行组件设计。
2. **用户交互 (必须执行)**：
   - 运行 `scripts/phase3_confirmation.py` 管理四要素确认流。
   - 针对每个选定的模块，向用户逐一确认以下 4 点：
     1. **数据来源**：该组件取用第一阶段提炼的哪个业务模块的信息？（需同时确认该来源的数据完整性是否足以支撑该组件的图表渲染）
     2. **呈现方法**：组件内的视觉形态是否符合预期？
     3. **回报内容**：组件内具体需要展示哪些数据字段？
     4. **下钻逻辑**：点击该组件后，跳转关联到哪个详细模块？
   - 处理异常分支：
     - **异常分支 E**（数据来源不明确）：返回阶段一补充数据字段定义，通过 `resume_from_branch_e()` 恢复。
     - **异常分支 F**（呈现方法需调整）：真正调用备选组件列表，用户可选择切换备选组件（输入编号）或保留自定义调整（输入「自定义」），通过 `_handle_branch_f_selection()` 闭环，再通过 `resume_from_branch_f()` 恢复。
     - **异常分支 G**（下钻逻辑缺失）：标注「暂不下钻」并继续流程。
   - 等待用户确认所有组件的细节。
3. **生成代码**：
   - 在用户确认所有细节后，**必须先读取 `references/design_system.md`**，了解 iOS 26 Liquid Glass 视觉风格的完整设计 Token（颜色、字体、间距、圆角、阴影、动效、毛玻璃效果），确保生成的代码严格遵循该设计规范。
   - 运行 `scripts/code_generator.py`，读取 `templates/dashboard_template.tsx` 作为参考，根据确认的组件选型、数据绑定和下钻逻辑，生成最终的 React/TypeScript 周报网页代码。
   - 最终输出完整可运行的周报网页单文件（如 `templates/weekly_report_example.tsx` 所示）。
   - 如果用户验收代码时需要修改，支持针对特定组件或样式的精准修改，循环至验收通过。

## 资源文件说明

- `references/analysis_framework.md`: 业务模块分析框架，用于第一阶段的文档解析。
- `references/component_library.md`: 组件库规范（预设 **15 个组件**的注册结构，含口径说明浮层 `metric_definition_popover`），用于第二、三阶段的模块提炼和组件选型。
- `references/hero_component_design_principles.md`: **首屏组件设计规范**，定义"一眼定调 → 悬浮概要/关键 → 下钻详情"三层交互模型的详细原则与检查清单。**特别强调了高密度结构化悬浮卡片的设计规范（包含状态、风险、洞察等多视角组合）及智能边缘防遮挡逻辑。当需要创建预设组件以外的新首屏组件时，必须优先读取本文件。**
- `references/component_brainstorm.md`: 所有 **15 个组件**（含已实现 5 个 + 待扩展 10 个 + 口径说明浮层 1 个）的完整设计方案，包含每个组件的三层交互设计要点，供实现时参考。
- `references/design_system.md`: **iOS 26 Liquid Glass 设计规范**，包含完整的 Design Token 系统（颜色/字体/间距/圆角/阴影/动效/毛玻璃效果），覆盖 14 个维度，含附录 Token JSON。**阶段 3 生成代码前必须读取本文件。**
- `templates/dashboard_template.tsx`: 周报仪表盘 React 组件模板，用于最终生成网页代码。
- `templates/weekly_report_example.tsx`: 完整可运行的周报网页示例输出。
- `scripts/`: 包含各阶段核心逻辑的 Python 脚本。
