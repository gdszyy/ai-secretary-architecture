在执行周报构建的第二阶段（模块提炼与优先级划分）和第三阶段（组件选型与数据绑定）时，必须参考本规范。本规范定义了周报模块的分类、核心 UI 组件的选型标准以及与用户交互的确认要素。

## 1. 模块分类与呈现优先级

周报的呈现结构分为两大类：**首屏模块 (Hero Modules)** 和 **其他模块 (Secondary Modules)**。这种分层设计旨在确保核心信息能够第一时间触达读者，同时提供足够的深度以供下钻分析。

### 1.1 首屏模块 (Hero Modules) - 优先级最高 (P0)

首屏模块必须在用户打开周报的第一眼呈现，用于快速传递项目的整体健康度和核心指标。这些模块是整个周报的“门面”，决定了读者对项目当前状态的第一印象。

| 模块名称 | 核心定义与作用 |
| :--- | :--- |
| **项目健康度诊断 (Health Score)** | 综合评分，通常包含增长、产品、商业化、风险等维度的雷达图或评分卡。该模块需要直观地反映项目的整体运行状况。 |
| **北极星指标归因 (North Star Attribution)** | 核心指标的当前值、环比/同比变化，以及导致该变化的主要因素拆解。这是评估项目是否在正确轨道上的关键。 |
| **KPI 监控与数据异常 (KPI Monitor & Anomalies)** | 关键业务指标的红绿灯状态，特别是突破阈值的异常告警。该模块用于快速识别需要立即关注的问题领域。 |
| **里程碑进度 (Milestone Progress)** | 当前版本的核心目标达成情况，以及下一阶段的规划。这有助于对齐团队目标和预期。 |

### 1.2 其他模块 (Secondary Modules) - 优先级中等 (P1/P2)

其他模块通常通过首屏模块的下钻 (Drill-down) 或金刚区 (Navigation Bar) 跳转访问，提供更详细的业务数据和运营情况。这些模块为首屏的高层次指标提供了必要的上下文和细节支撑。

| 模块名称 | 核心定义与作用 |
| :--- | :--- |
| **业务模块详情 (Module Details)** | 各子业务线（如游戏、社交、商业化）的具体进展、核心数据和近期动作。 |
| **团队构成与动态 (Team & Organization)** | 人员变动、招聘需求、团队绩效。 |
| **财务情况 (Financial Status)** | 收入结构、成本支出、ROI 分析、现金流消耗。 |
| **风险与问题登记册 (Risk & Issue Register)** | 当前面临的阻塞问题、潜在风险点及应对策略。 |
| **决策与待办事项 (Decisions & Todos)** | 需要高层决策的事项列表，以及各模块的待办任务追踪。 |

## 2. 全量 UI 组件选型规范

在第三阶段，当用户选定需要呈现的模块后，必须为每个模块匹配合适的 UI 组件。以下是全量 14 个 UI 组件的选型规范。组件分为两类：

- **首屏组件**：遵循“一眼定调 → 悬浮概要 → 下钒详情”三层交互模型，详见 `references/hero_component_design_principles.md`。
- **详情组件**：直观清晰地把事情讲清楚，无三层交互限制。

### 2.1 首屏组件 (Hero Components) - P0

| 组件名称 | 组件 Key | 适用模块 | 一眼定调 | 悬浮概要 | 下钒详情 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **评分卡组件** | `scorecard` | 项目健康度诊断、KPI 监控 | 巨大数值 + 迷你趋势线 + 背景色健康度 | **高密度悬浮卡**：同环比变化率、目标达成率进度条、Top 3 驱动因素、业务洞察 | 跳转至该指标的历史趋势图或归因分析 |
| **归因瀑布图/树状图组件** | `attribution_chart` | 北极星指标归因 | 瀑布图，绿柱=正向贡献，红柱=负向贡献 | **归因详情卡**：子维度变化绝对值、贡献百分比、业务动作说明、Top 1 异常指标预警 | 跳转至该子维度对应的业务模块详情 |
| **时间轴组件** | `timeline` | 里程碑进度 | 节点连线，延期节点标红+警告图标 | **里程碑快照卡**：计划/实际时间对比、关键交付物状态、延期阻塞原因及解决进度、核心参与人 | 跳转至详细的项目管理看板或需求文档 |
| **异常告警列表组件** | `anomaly_alert_list` | KPI 监控与数据异常、风险登记册 | 精简列表，P0 呼吸闪烁动效 | **结构化风险卡**：风险描述、发生时间线、受影响范围、当前应对策略、责任人头像 | 跳转至排查工单或责任人对话框 |
| **北极星因子分解图** | `north_star_decomposition` | 北极星指标归因 | 树状图，节点颜色健康度，连线粗细=权重 | **因子洞察卡**：因子当前值、目标差距、负责团队、数据深层洞察 (Data Insight) | 点击因子节点跳转至对应底层业务模块 |
| **多维健康度雷达图** | `health_radar_chart` | 项目健康度诊断 | 多边形雷达图，当前vs目标双层，警戒区域红色遮罩 | **维度诊断卡**：维度得分、阈值/目标进度条、核心支撑指标列表、主要扣分项列表 | 跳转至该维度对应的详细分析模块 |
| **业务阶段进度组件** | `stage_progress_bar` | 里程碑进度 | 横向步骤条，当前阶段高亮+整体进度百分比 | **阶段全景卡**：阶段副标题、核心目标、当前完成情况、2-3 个关键指标 (当前 vs 目标) | 跳转至该阶段详细规划文档或 OKR 看板 |

### 2.2 详情组件 (Detail Components) - P1

| 组件名称 | 组件 Key | 适用模块 | 设计方案与业务价值 |
| :--- | :--- | :--- | :--- |
| **模块进度卡片组件** | `module_progress_card` | 业务模块详情概览 | 网格布局卡片，包含负责人头像、整体进度条、近期核心动作列表。直观展示各条线的齐头并进情况。 |
| **漏斗转化组件** | `conversion_funnel` | 增长与获客 | 倒三角漏斗图或阶梯柱状图，标注每步绝对人数、步骤间转化率和整体转化率。支持多漏斗对比，直观暴露转化瓶颈。 |
| **指标趋势矩阵** | `trend_matrix_grid` | 多指标监控、业务模块详情 | 小倍数图（Small Multiples），多个 KPI 趋势图网格排列，共享 X 轴，鼠标悬浮时出现联动十字准星，便于横向比对。 |
| **用户分层气泡图** | `user_segmentation_bubble` | 产品与活跃、用户运营 | 四象限气泡图，X/Y 轴为两个核心评估维度，气泡大小=用户规模。通过象限划分直观展示用户价值分层，指导精细化运营。 |
| **创作者/供给方生态图** | `ecosystem_health_map` | 内容与社区、平台生态 | 金字塔图或双向堆叠柱状图，对比头/腰/尾部的人数规模占比与价值贡献占比，直观反映生态集中度，预警固化风险。 |
| **单元经济模型卡片** | `unit_economics_card` | 商业化与变现、财务情况 | 高密度数据面板，核心突出 LTV/CAC 比值，下方以公式树形式拆解 LTV 和 CAC 的构成要素，支持展开查看底层变量趋势。 |
| **里程碑卡片** | `milestone_card_list` | 里程碑进度 | 竖直滚动面板，每行一个里程碑卡片，包含状态标签、计划/实际时间对比、负责人、进度条等。支持按状态筛选，延期项高亮。 |

## 3. 第三阶段交互的 4 要素确认规范

在向用户确认组件选型时，必须明确询问并确认以下 4 个核心要素，以确保最终生成的周报网页符合用户预期：

| 确认要素 | 核心问题与示例 |
| :--- | :--- |
| **数据来源** | 该组件的数据从第一阶段提炼的哪个业务模块中获取？（例如：DAU 数据来源于“增长与获客模块”）。 |
| **呈现方法** | 确认组件的视觉形态是否符合预期？（例如：是否需要趋势图？是否需要红绿灯告警？是否需要特定的颜色映射？）。 |
| **回报内容** | 组件内具体需要展示哪些数据字段？（例如：除了当前值，是否还需要展示同比/环比变化率、目标达成率等？）。 |
| **下钻逻辑** | 用户点击该组件后，期望跳转到哪个关联模块查看更详细的信息？（例如：点击“商业化健康度”评分卡，跳转到“财务情况”模块）。 |
