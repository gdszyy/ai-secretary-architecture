# ai-secretary-architecture 仓库迭代评估报告

基于“项目管理文档即项目本身”的设计理念，本报告对 `ai-secretary-architecture` 仓库的当前完成状态进行了深度评估。通过对照三层架构模型，结合 XP 平台项目概要与生命周期图，识别了已完成内容与待完成目标，并提出了具体的迭代改进方案。

## 一、 现状评估：对照三层架构的完成度

根据《“项目管理文档即项目本身”：模块化沉淀与自进化架构设计》文档，系统应具备三个层次的能力。以下是当前仓库的实现情况评估：

### 1. 第一层：双轨跟进机制（被动聚合与主动定调）

**完成度：较高 (约 80%)**

*   **已完成内容**：
    *   **被动信息聚合**：已实现 `scripts/daily_progress_updater.py`，能够通过游标机制（`data/fetch_cursor.json`）定时拉取飞书群聊消息，并利用 LLM 提取各模块的进展更新。
    *   **意图分类体系**：已建立 `major_decision`、`milestone_fact`、`risk_blocker`、`routine_task` 等意图分类，并将高价值事实写入 Bitable 话题表。
    *   **自动化批处理**：已实现 `scripts/run_daily_batch.py` 和 `scripts/run_weekly_batch.py`，形成了每日增量拉取和每周汇总的自动化闭环。
*   **待完成目标**：
    *   **主动定调闭环缺失**：目前缺乏负责人在飞书群内直接回复卡片进行定调和纠正的闭环。根据 `docs/module2_buffer/lark_card_reply_correction_spec.md`，PF-001 需求（飞书卡片回复自动触发信息纠正）仍处于待开发状态。
    *   **多源聚合不足**：目前主要依赖飞书群聊，对代码仓库（Commit/PR）和设计稿更新动态的自动抓取尚未完全实现。

### 2. 第二层：结构性诊断与里程碑倒推

**完成度：中等 (约 50%)**

*   **已完成内容**：
    *   **风险预警**：已实现 `scripts/weekly_issue_reminder.py`，每周四自动扫描本周的 `risk_blocker` 并发送飞书提醒卡片。
    *   **里程碑数据结构**：`data/dashboard_data.json` 中已包含 `milestones` 字段，记录了项目的硬性里程碑及其状态（如 `due_date`、`status`、`progress`）。
    *   **全局里程碑补充**：`scripts/enrich_global_milestones.py` 能够为全局里程碑补充模块快照并检测延期。
*   **待完成目标**：
    *   **结构性失衡诊断**：目前缺乏跨模块健康度的交叉比对机制。例如，无法自动识别“引流获客模块超前但国别配置滞后”的结构性风险。
    *   **动态优先级与倒推计算**：尚未实现基于硬性里程碑的剩余时间缓冲（Buffer）自动倒推计算，以及基于此的动态优先级调整。
    *   **团队负载与风险蔓延预测**：缺乏团队负载热力图生成能力，且无法自动评估底层模块延期对上层业务的蔓延影响。

### 3. 第三层：活体产品文档沉淀（功能现状与迭代累积）

**完成度：初级 (约 30%)**

*   **已完成内容**：
    *   **项目数字孪生雏形**：`data/dashboard_data.json` 作为项目的“数字孪生”，已结构化地记录了模块分组、KPI 指标、里程碑和各模块的每周动态。
    *   **活文档契约规范**：在 `AGENTS.md` 和 `global.md` 中明确规定了代码修改必须同步更新 `.cursor/rules/` 下的规范文档，并沉淀流程洞察。
*   **待完成目标**：
    *   **业务全景映射不全**：对比《XP平台项目概要》和 `xp_lifecycle.mmd`，当前 `dashboard_data.json` 中的 12 个模块未能完全覆盖 XP 平台的 9 大阶段（如 CMS 内容管理、留存激励中的复杂礼券系统等细分模块映射不够清晰）。
    *   **设计意图与现状维护缺失**：目前缺乏强制在文档头部包含“设计上下文（Design Context）”的机制，且功能点矩阵和核心算法逻辑流的自动沉淀与更新尚未实现。
    *   **迭代累积的自动化**：尚未实现将 Jira/Meegle 中的迭代记录自动回溯并覆写到原本文档沉淀中的功能。

## 二、 结合 XP 平台业务的缺口识别

通过对比《XP平台项目概要》和 `xp_lifecycle.mmd`，当前 AI 秘书系统在业务映射上存在以下缺口：

1.  **国别与站点配置（阶段 2）**：缺乏对租户管理、IP 限制、支付路由等细粒度配置的独立追踪模块。
2.  **实名认证 KYC（阶段 4）**：三方 KYC 接入的进度和风险追踪在当前模块划分中不够突出。
3.  **CMS 内容管理（阶段 5）**：盘口热门算法、首页推荐配置等核心运营支撑能力未在 Dashboard 中独立体现。
4.  **核心业务（阶段 6）**：体育博彩的风控系统（MTS/Defend）和结算模块逻辑复杂，当前仅作为“体育博彩核心”的子项，缺乏深度的状态追踪。
5.  **留存激励（阶段 7）**：复杂的礼券系统（5种类型、降落伞机制）和预算池管理需要更精细的模块划分和进度监控。

## 三、 迭代改进方案与目标

为了实现“项目管理文档即项目本身”的终极目标，建议按以下阶段进行迭代：

### 阶段一：补齐双轨跟进闭环（近期目标）

1.  **实现 PF-001 需求**：开发 `scripts/lark_webhook_server.py`，监听飞书消息事件，支持负责人通过回复卡片（`纠正：...` 或 `补充：...`）直接更新 Bitable 和 Dashboard。
2.  **扩展信息源**：接入 GitHub Webhook，自动捕获代码提交和 PR 动态，丰富被动信息聚合的维度。

### 阶段二：深化结构性诊断与业务映射（中期目标）

1.  **重构模块映射**：根据 `xp_lifecycle.mmd`，细化 `dashboard_data.json` 中的模块划分，确保覆盖 CMS、礼券系统、风控结算等核心业务节点。
2.  **开发诊断引擎**：在 `run_weekly_batch.py` 中引入结构性诊断逻辑，交叉比对强依赖模块的进度，自动生成风险蔓延预警。
3.  **实现里程碑倒推**：基于 `due_date` 自动计算 Buffer，并在 Dashboard 中高亮显示 Buffer 告急的模块。

### 阶段三：实现活体文档的自动演进（远期目标）

1.  **设计上下文强制校验**：在 Agent 修改代码或文档时，强制要求提取并更新“设计意图”和“历史困境”。
2.  **功能现状自动覆写**：开发文档解析与更新脚本，当 Meegle 中的 Story/Defect 完成时，自动将变更内容（Changelog）追加到对应模块的活体文档中，并更新功能点矩阵。
3.  **团队负载可视化**：整合 Meegle 任务分配数据，在前端看板中生成团队负载热力图，识别单点故障。

## 四、 结论

`ai-secretary-architecture` 仓库已经成功搭建了“项目数字孪生”的底层数据结构和自动化的信息聚合跑批流程，实现了从“记录工具”向“动态看板”的跨越。然而，在主动定调闭环、深度的结构性诊断以及活体文档的自动演进方面仍有较大提升空间。通过实施上述迭代方案，系统将能够真正实现“文档即项目本身”的自进化管理生态。
