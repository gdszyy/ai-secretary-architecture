# 项目里程碑管理规范 (Milestone Policy)

## 1. 核心原则

**里程碑必须由项目管理人员手动设定，严禁 AI 自动生成。**

AI 秘书系统在处理项目进度时，仅负责**追踪**和**诊断**里程碑的达成情况，而不能**创造**里程碑。里程碑是项目战略方向的锚点，代表了业务上的关键节点（如：某国别牌照生效日、大型体育赛事开赛日、核心系统上线日），必须由人类决策者（项目经理/产品负责人）进行规划和确认。

## 2. 里程碑数据结构

在 `data/dashboard_data.json` 中，里程碑数据存储在 `milestones` 数组中。人工设定里程碑时，必须遵循以下数据结构：

```json
{
  "id": "ms_2026_q2_launch",
  "title": "XP-BET 平台 V1.0 生产环境上线",
  "due_date": "2026-05-15",
  "status": "pending",
  "group_id": "grp_platform",
  "description": "包含体育博彩核心、基础 Casino 接入及首充活动，支持巴西市场。"
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | String | 是 | 里程碑唯一标识符，建议使用 `ms_` 前缀加描述性名称。 |
| `title` | String | 是 | 里程碑名称，需简明扼要。 |
| `due_date` | String | 是 | 截止日期，格式为 `YYYY-MM-DD`。这是计算 Buffer 的关键依据。 |
| `status` | Enum | 是 | 当前状态。可选值：`pending` (未开始), `in_progress` (进行中), `completed` (已完成), `overdue` (已延期)。 |
| `group_id` | String | 否 | 关联的模块分组 ID（如 `grp_sports`）。如果属于全局里程碑，可留空。 |
| `description` | String | 否 | 里程碑的详细描述或验收标准。 |

## 3. AI 秘书的职责边界

在里程碑管理中，AI 秘书（如 `run_weekly_batch.py` 等自动化脚本）的职责严格限制在以下范围：

1.  **状态监控与预警**：定期检查 `due_date`，如果当前日期超过 `due_date` 且状态不为 `completed`，AI 秘书有权将状态自动更新为 `overdue`，并触发延期预警。
2.  **Buffer 倒推计算**：根据当前日期与 `due_date` 的距离，计算剩余缓冲时间（Buffer），并在看板中高亮显示风险。
3.  **进度快照关联**：在里程碑到达时，自动截取相关模块的进度快照（Snapshot）并关联到该里程碑下，作为历史归档。

**禁止行为**：
*   AI 秘书**不得**自动向 `milestones` 数组中添加新的里程碑记录。
*   AI 秘书**不得**自动修改人工设定的 `due_date`。
*   AI 秘书**不得**自动删除未完成的里程碑。

## 4. 设定流程

1.  项目经理在项目规划阶段或迭代计划会议上确定关键里程碑。
2.  项目经理通过修改 `data/dashboard_data.json` 文件（或通过未来的前端管理界面）手动录入里程碑数据。
3.  AI 秘书系统在下一次跑批时自动识别并开始追踪新设定的里程碑。
