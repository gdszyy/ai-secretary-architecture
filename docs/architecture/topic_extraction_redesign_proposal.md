# 话题提取与价值判断重构设计提案 (Topic Extraction Redesign Proposal)

**作者**: Manus AI
**日期**: 2026-04-20
**关联模块**: `module2_buffer`, `module3_info_sources`

---

## 1. 背景与核心痛点

在当前的 AI 项目秘书架构中，信息缓冲池（Module 2）和群消息解析方案（Module 3）倾向于将群聊中的所有讨论视为潜在的“待跟进事项”，并设计了复杂的“主动追问”和“未闭环话题追踪”机制 [1] [2]。

然而，根据最新的业务反馈，这种设计存在严重的**价值错位**：
1. **性价比极低**：大部分群聊话题（如日常沟通、小问题汇报）在讨论后已经有了解决方案，或者会自然流转到 Meegle 处理，并不具备持续追踪的价值。
2. **核心价值偏离**：话题提取的真正价值在于**信息沉淀与真实性校验**，即从群聊中提取出里程碑、商务三方选择、重大决策等高价值信息，作为项目记录的节点和进度依据。
3. **角色混淆**：系统过度扮演了“催办员”的角色，而忽视了“项目档案馆”的核心定位。对于特定人员（如 VoidZ）的个人跟进，属于“个人秘书”范畴，不应与“项目级追踪”混为一谈。

基于上述痛点，本提案旨在对系统的话题提取逻辑和价值判断标准进行根本性重构。

---

## 2. 核心设计调整方向

### 2.1 价值判断标准重塑：从“跟进驱动”转向“事实沉淀驱动”

系统必须放弃“凡事皆需闭环”的执念，将群聊定位为**高价值信号的提取源**，而非**待办事项的生成器**。

**调整策略**：
*   **大幅削减“备忘 (Memo)”和“进度更新 (Progress Update)”的跟进权重**：除非明确提及了核心模块的重大延期，否则不再主动追问。
*   **强化“决策记录 (Decision Record)”和“风险升级 (Risk Escalation)”的提取**：将群聊中的商务选择（如 Lsport 对接）、技术选型、里程碑确认作为最高优先级提取对象 [3]。
*   **引入“事实校验 (Truth Validation)”机制**：提取到的进度或决策，必须与结构化系统（如 Meegle、Lark 看板）进行交叉比对，确认其真实性后，再作为项目节点记录。

### 2.2 意图分类体系精简与重构

在 `info_buffer_design.md` 和 `prereq_data_assessment.md` 的基础上，对意图分类进行以下调整 [4] [5]：

| 调整后意图类型 | 核心价值定位 | 提取策略与下游流转 |
| :--- | :--- | :--- |
| **重大决策 (Major Decision)** | **极高** (如：商务三方选择、架构拍板) | 提取决策背景、结论和决策人。直接归档至 Lark Wiki 或项目决策记录表，**不触发追问**。 |
| **里程碑/进度事实 (Milestone Fact)** | **高** (如：某模块提测、上线) | 提取进度节点。与 Meegle 状态交叉比对，若一致则更新看板；若不一致则记录为“待核实进度”，**仅在看板层面提示 PM**。 |
| **风险/阻塞 (Blocker/Risk)** | **高** (如：SR商务不配合、牌照问题) | 提取风险点和影响范围。推入风险池，若涉及外部依赖，可触发一次确认。 |
| **常规缺陷/需求 (Routine Task)** | **低** (项目级) | 提取后直接尝试映射到 Meegle。若信息不足，**直接丢弃或降级为日志**，不再主动在群内追问（除非明确 @AI秘书）。 |
| **个人跟进 (Personal Follow-up)** | **中** (仅限特定人员，如 VoidZ) | 仅当话题明确涉及特定人员的待办时，作为“个人秘书”功能推送到其私人待办列表，**不进入项目全局看板**。 |

### 2.3 群消息解析流水线 (Pipeline) 改造

针对 `lark_group_message_reading_plan_v2.md` 中的流水线，进行以下“降噪”改造 [2]：

1.  **取消“未闭环话题 (Pending Threads)”的强制提升调度**：不再因为某个小 Bug 没讨论完就频繁拉取群消息。
2.  **重写“智能追问 (Inquire)”逻辑**：
    *   **废弃**：针对常规 Bug、日常进度的群内自动 @ 追问。
    *   **保留**：仅对“重大决策”缺失关键上下文，或“风险/阻塞”需要明确责任人时，才触发追问。
3.  **强化“群组画像”的上下文作用**：利用群名（如“前端组”、“数据对接组”）作为强上下文，辅助 LLM 判断话题的业务归属，而非仅仅依赖内容提取。

## 3. 业务话题补充与系统映射示例

根据最新的业务话题补充，系统应能自动提取并映射为以下结构化记录：

### 3.1 平台统一赔率限制
*   **原始信息**：活动平台统一使用相同的 Bonus 赔率限制，避免主钱包能量分配漏洞。
*   **系统处理**：识别为 **重大决策 (Major Decision)**。
*   **流转**：提取“Bonus 赔率限制统一”作为决策点，归档至 `活动系统` 的架构决策记录中。

### 3.2 多语言翻译确认
*   **原始信息**：建立多语言对照表格，pending @tao 给 @VoidZ 前端翻译 js 文件。
*   **系统处理**：识别为 **个人跟进 (Personal Follow-up)**。
*   **流转**：不进入项目主看板。推送到 VoidZ 的个人待办/提醒列表中。

### 3.3 体育数据供应商选择
*   **原始信息**：SR商务不配合，换新公司对接；积极接触其他供应商，目前对接 Lsport；牌照要求高导致供应商不积极。
*   **系统处理**：识别为 **风险/阻塞 (Blocker/Risk)** 与 **重大决策 (Major Decision)** 的复合体。
*   **流转**：记录“SR 商务替换”和“对接 Lsport”为关键节点；将“牌照进度影响供应商对接”标记为项目级风险。

### 3.4 CRM 系统选型
*   **原始信息**：要求接入 Smartico，因为集团多数使用了这个平台。
*   **系统处理**：识别为 **重大决策 (Major Decision)**。
*   **流转**：归档至 `CRM系统` 的选型决策记录中，并标记原因“集团统一技术栈”。

## 4. 实施建议与下一步行动

1.  **更新 Buffer 状态机**：修改 `module2_buffer` 的代码逻辑，大幅降低常规意图的 `completeness_score` 阈值要求，允许“不完整”的常规话题直接归档或丢弃，切断追问循环。
2.  **重写 LLM Prompt**：在群消息解析的 Prompt 中，明确注入“你是项目档案馆，不是催办员”的系统设定，强调对决策、里程碑和风险的提取。
3.  **分离个人秘书逻辑**：在架构上将“项目级看板同步”与“个人级待办提醒”解耦，确保类似“@tao 给 @VoidZ 翻译文件”的事务不会污染项目主线。
