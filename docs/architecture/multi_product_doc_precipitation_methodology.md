# 多产品文档提炼与沉淀方法论 (Multi-Product Documentation Precipitation Methodology)

**作者**: Manus AI
**日期**: 2026-04-22

## 1. 背景与痛点分析

在多产品线并行的研发环境中，文档管理通常面临以下核心痛点：

*   **信息孤岛与碎片化**：多个产品的文档散落在各自的飞书云文档、Wiki 或独立仓库中，缺乏统一的检索入口和全局视角。
*   **缺乏抽象与复用**：各产品线独立维护相似的功能模块（如用户认证、支付路由、活动中心），功能点未被提炼为可复用的组件或业务模型，导致重复造轮子。
*   **状态脱节与同步困难**：日常的小型修改、Bug 修复或临时决策往往直接在代码或零散的沟通记录中发生，无法及时同步到他人的文档中，导致“代码即文档”的契约破裂。
*   **AI Agent 上下文污染**：当 AI Agent 介入时，如果仓库中混合了大量过程性材料、历史任务记录和未提炼的原始数据，会严重降低 Agent 的理解效率和行动准确率。

## 2. 核心理念：从“文档驱动”到“模型驱动”

为了解决上述痛点，我们需要建立一套提炼抽象的方法流程，将文档的定位从“静态的说明书”转变为“动态的业务模型契约”。

*   **Single Source of Truth (SSOT)**：确立 Git 仓库作为核心架构、业务模型和规范的唯一事实来源。飞书云文档等外部系统应作为展示层或协作草稿区，最终的结构化知识必须沉淀回仓库。
*   **分层抽象体系**：将文档分为“全局规范层”、“业务模型层（提炼后）”和“产品实例层”。
*   **活文档契约 (Living Documentation)**：通过自动化工具（如 `code-indexer`、`repo-navigator`）和严格的 SOP，确保代码修改与文档更新的强绑定。

## 3. 文档沉淀与提炼方法流程

### 3.1 第一阶段：信息捕获与标准化 (Capture & Standardize)

*   **统一入口**：利用 AI 秘书（如 `module2_buffer`）统一接收来自各渠道（飞书群、会议纪要、零散需求）的碎片化信息。
*   **结构化转化**：通过 LLM 将碎片信息转化为标准的工作项（Story、Defect、Task），并录入到统一的看板系统（如 Meegle 或飞书多维表格）。
*   **草稿沉淀**：在飞书云文档中进行初步的需求讨论和草稿编写。

### 3.2 第二阶段：提炼与抽象 (Extract & Abstract)

这是最关键的一步，旨在从具体产品的需求中提取通用的业务逻辑和功能点。

*   **功能点映射 (Feature Mapping)**：建立全局的“功能地图”。当一个新产品提出需求时，首先在功能地图中寻找是否已有类似功能点。
*   **业务模型提炼**：对于跨产品复用的功能（如“钱包划转”），将其提炼为独立的业务模型文档（如 `PRD_Wallet_Core.md`），剥离特定产品的 UI/UX 细节。
*   **建立双向索引**：采用“混合矩阵模式（Hybrid Matrix Model）”。将提炼出的核心业务逻辑（Why+What）与具体产品的交互规范（How）分离，并通过功能点 ID 建立双向锚点链接。

### 3.3 第三阶段：入库与持久化 (Precipitate & Persist)

*   **Markdown 转化**：将飞书云文档中定稿的内容，通过自动化脚本（或人工确认）转化为标准的 Markdown 格式。
*   **分仓存储策略**：遵循 `repository_split_assessment.md` 的原则，避免 Monorepo 反模式：
    *   `architecture-repo`：存储全局规范、提炼后的核心业务模型、API 契约。
    *   `product-a-repo` / `product-b-repo`：存储特定产品的 UI/UX 规范、前端代码和具体实例。
*   **流程洞察沉淀 (Process Insights)**：在开发过程中发现的隐蔽逻辑、跨模块耦合陷阱，必须提炼为 `PI-xxx.md` 洞察文档，存入 `.cursor/rules/process_insights/`，供后续 Agent 或开发者避坑。

### 3.4 第四阶段：自动化同步与治理 (Sync & Govern)

*   **代码-文档同步契约**：任何对核心逻辑的修改，必须在同一个 Git Commit 中同步更新对应的 `.cursor/rules/*.md` 规范文档。
*   **自动索引更新**：利用 `code-indexer` 自动扫描代码变更，更新函数级索引（`auto_index`），确保 Agent 导航的准确性。
*   **状态回写**：利用 `lark-md-import` 和 `lark-wiki-docs` 等技能，将仓库中更新后的 Markdown 文档自动同步回飞书云文档，保持展示层的最新状态。

## 4. 实施建议与下一步行动

1.  **盘点现有资产**：对各产品线的现有云文档进行一次全面盘点，识别出高频复用的功能模块。
2.  **建立全局功能地图**：在架构仓库中初始化一份全局功能地图（如 `global_feature_map.md`），为每个核心功能分配唯一 ID。
3.  **试点重构**：选择一个跨产品复用度高的模块（如“用户认证”或“支付”），按照上述方法论进行提炼，剥离出核心业务模型并沉淀到架构仓库。
4.  **完善自动化工具链**：确保 `repo-navigator`、`code-indexer` 和飞书同步脚本的稳定运行，降低人工维护文档的成本。
