# PRD 大功能-小功能层级存储与索引规范

**作者**: Manus AI
**日期**: 2026-04-22

## 1. 核心结论：双层存储模型

产品需求文档（PRD）沉淀后，**必然且必须**变成“大功能-小功能”的结构。但这种结构**不应该**完全通过“文件夹嵌套文件夹”或“一个几万字的巨型 Markdown 文件”来实现。

结合本仓库中已有的实践（如 `sportsbook-recommendation` 的目录结构、`module1_kanban` 的多维表格设计），最佳的存储方式是**双层存储模型**：

1.  **文档层（大功能）**：以“大功能/业务模块”为粒度，沉淀为独立的 Markdown 文件（如 `PRD_Wallet.md`）。
2.  **数据层（小功能）**：以“小功能/具体需求点”为粒度，沉淀为结构化的数据表（如飞书多维表格中的“功能表”），并通过元数据（Metadata）与文档层建立双向链接。

## 2. 为什么不建议“全文档化”小功能？

如果把每一个小功能（如“钱包余额隐藏按钮”、“密码输入框防抖”）都写成独立的 Markdown 文件，或者全部塞进一个巨型 PRD 中，会导致以下问题：

*   **文件碎片化**：仓库中会出现数百个极小的 `.md` 文件，难以维护和阅读。
*   **上下文过载**：巨型 PRD 会导致 AI Agent 读取时消耗大量 Token，且极易产生幻觉（正如 `repository_split_assessment.md` 中指出的痛点）。
*   **状态追踪困难**：小功能的生命周期（待开发、测试中、已上线）变化极快，Markdown 文件不适合高频的状态更新。

## 3. 具体存储方案设计

### 3.1 目录结构：以“大功能”为文件边界

在产品需求仓库（如 `crm-requirements` 或 `xpbet-frontend-requirements`）中，目录结构应保持扁平，以大功能模块划分文件。

```text
docs/
├── README.md                  # 产品的全局索引与功能地图
├── PRD_01_UserAuth.md         # 大功能：用户认证系统
├── PRD_02_Wallet.md           # 大功能：钱包与资金系统
├── PRD_03_ActivityCenter.md   # 大功能：活动中心
└── UX_01_UserAuth.md          # 对应的交互规范（混合矩阵模式）
```

**参考案例**：在 `docs/sportsbook-recommendation/README.md` 中，推荐系统被拆分为 8 份核心 PRD（如 `PRD-03 首页比赛推荐`），而不是把所有细节写在一个文件里，也没有为每个小推荐位建独立文件。

### 3.2 文档内部：小功能作为章节（Section）

在 `PRD_02_Wallet.md` 内部，小功能作为二级或三级标题存在。

```markdown
# PRD_02 钱包与资金系统

## 1. 业务概述
...

## 2. 功能详情

### 2.1 主钱包划转 (Feature ID: FP-WALLET-001)
- **需求描述**：...
- **业务规则**：...

### 2.2 余额隐藏开关 (Feature ID: FP-WALLET-002)
- **需求描述**：...
```

**关键约束**：每个小功能必须拥有一个全局唯一的 **Feature ID**。

### 3.3 结构化数据层：小功能的生命周期管理

小功能的流转状态、排期、负责人等动态信息，**不存放在 Markdown 中**，而是存放在飞书多维表格（或 Meegle 等项目管理工具）的“功能表”中。

根据 `docs/module1_kanban/lark_kanban_design.md` 的设计，功能表包含：

| Feature ID | 功能名称 | 所属模块 (大功能) | 状态 | 文档链接 |
| :--- | :--- | :--- | :--- | :--- |
| FP-WALLET-001 | 主钱包划转 | 钱包与资金系统 | 已上线 | `[PRD_02_Wallet.md#21-主钱包划转](...)` |
| FP-WALLET-002 | 余额隐藏开关 | 钱包与资金系统 | 开发中 | `[PRD_02_Wallet.md#22-余额隐藏开关](...)` |

### 3.4 索引层：README 作为全局路由

每个产品目录下的 `README.md` 充当该产品的“功能地图”和路由表。它列出所有大功能，并简述其包含的小功能。

## 4. 总结

*   **大功能** = 静态的 Markdown 文件（PRD）。
*   **小功能** = PRD 内部的章节（带 Feature ID） + 多维表格中的一条动态记录。
*   **连接纽带** = Feature ID 与精准的文档锚点链接。

这种结构既保证了业务逻辑的完整性（大功能文档），又满足了项目管理的灵活性（小功能数据条目），同时对 AI Agent 极其友好（通过 Feature ID 精准定位）。
