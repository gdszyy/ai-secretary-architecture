# AI 秘书机器人介入场景评估与设计方案

**作者**: Manus AI
**日期**: 2026-04-15
**所属模块**: 模块二 (信息缓冲池)

---

## 1. 评估背景与目标

在当前的 AI Management 项目秘书系统中，AI 机器人通过被动监听或主动拉取的方式获取群聊消息。然而，在日常的高频讨论中，尤其是涉及技术前端和 Bug 修复的场景，AI 仅仅作为被动的记录者无法最大化其价值。

为了进一步提升项目管理的效率，减轻项目经理的事务性负担，我们需要明确在何种场景下，AI 机器人可以主动拦截对话并直接向相关人员发起询问。本评估报告基于 `lark_group_weekly_report_0406_0410.md` 中的历史群聊数据，结合现有的意图识别与防堆积策略，设计了一套结构化的机器人介入场景评估框架，重点聚焦于前端技术讨论和 Bug 报送至 Meegle 的流程。

---

## 2. 机器人介入的核心原则

在设计介入场景前，必须确立机器人主动打断对话的核心原则，以平衡“及时性”与“免打扰”。

1. **高置信度前提**：机器人必须在完成多对话分离（Thread Separation）并获得高置信度（`confidence ≥ 0.8`）的独立上下文后，方可评估是否介入。
2. **价值导向**：介入必须能直接推动工作流（如：补全必填字段以创建 Meegle 工单），而非仅仅为了刷存在感。
3. **优先级分层**：对高优问题（如阻碍性 Bug）采用即时询问，对中低优问题（如日常优化建议）采用批量询问。
4. **降级机制**：若用户未响应，需有明确的降级归档策略，避免死锁。

## 3. 历史群聊场景分析与介入评估

通过分析历史群聊记录，我们提取了以下典型场景，并评估了机器人的介入价值。

### 3.1 场景一：前端缺陷报告与 Meegle 报送 (高优)

在“小组长”和“上线前前端优化需求”群组中，频繁出现关于前端 Bug 的讨论。

**历史样本**：
> "活动中心已基本验收，但投放后台存在较多问题，特别是 xp 技术未完成 ch 逻辑。"
> "发现游戏渲染加载较慢，VPN 连接速度可接受，直连速度较快。"

**场景特征**：
- **意图类型**：`Bug Report` 或 `Risk Escalation`。
- **信息缺失**：通常只有现象描述，缺乏具体的复现步骤、影响范围和优先级，导致无法直接转化为 Meegle 中的 Defect 工单。

**介入策略评估**：
- **触发条件**：LLM 识别为前端 `Bug Report`，且完整度评分 `< 80`（缺少复现步骤或影响范围）。
- **介入时机**：即时介入（Immediate Inquiry）。
- **介入目标**：收集足够的信息以自动在 Meegle 中创建 Defect 工单。
- **介入话术示例**：
  > **[紧急确认] 关于 前端渲染 的 Bug Report**
  > 收到关于“游戏渲染加载较慢”的反馈。为了尽快在 Meegle 创建工单交由研发处理，还需要补充：
  > 1. **复现步骤/环境**：(如：测试环境还是生产环境？具体机型？)
  > 2. **优先级**：(如：是否阻碍测试或上线？)
  > 请直接回复，我将立即为您建档。

### 3.2 场景二：前端功能需求与 UI 优化跟进 (中优)

在“设计稿优化对接”群组中，大量讨论涉及 UI 细节调整和交互逻辑的确认。

**历史样本**：
> "讨论了 H5 端 Tooltip 的统一处理方式，决定采用弹窗形式，并实现从问号图标弹出和收起的动画效果。"
> "移动端按钮高度保持首页/live/体育详情页为 56，outright/比赛详情页为 40。"

**场景特征**：
- **意图类型**：`Feature Request` 或 `Decision Record`。
- **信息缺失**：讨论往往产生决策，但缺乏明确的执行人（Assignee）和预期完成时间（Due Date），导致需求悬而未决。

**介入策略评估**：
- **触发条件**：识别到明确的 UI 优化决策（`Decision Record`）或新需求（`Feature Request`），但未提及责任人或时间节点。
- **介入时机**：批量询问（Batch Inquiry），如每日下班前汇总。
- **介入目标**：确认责任人和排期，将其转化为 Lark 看板的待办事项或 Meegle 的 Story。
- **介入话术示例**：
  > **[日常确认] 前端 UI 优化待办确认**
  > 关于“H5 端 Tooltip 弹窗动画实现”的决策，请问：
  > 1. 预期由哪位研发同学跟进？
  > 2. 期望的提测/上线时间是？

### 3.3 场景三：第三方数据源与接口联调阻塞 (高优)

在“Lsport数据源”群组中，经常涉及后端与第三方接口的联调问题，这些问题往往会阻塞前端的进度。

**历史样本**：
> "defend 的 WebSocket 连接存在断开问题，断开时客户端收到 'transport close websocket connection closed' 错误。"
> "lsport 数据中 competitor 缺少多语言（en, es, pt）数据，调用翻译接口结果为空。"

**场景特征**：
- **意图类型**：`Bug Report` 或 `Blocker`。
- **信息缺失**：通常涉及跨团队或外部供应商，缺乏明确的 Escalation 路径和跟进状态。

**介入策略评估**：
- **触发条件**：识别到接口报错或联调阻塞，且长时间（如 2 小时）未有明确解决方案。
- **介入时机**：即时介入，或在识别到阻塞状态后触发。
- **介入目标**：推动问题升级（Escalation），记录阻塞原因，并在 Meegle 中标记关联任务为 Blocked。
- **介入话术示例**：
  > **[阻塞确认] 关于 WebSocket 连接断开 问题**
  > 监测到该联调问题可能阻塞前端进度。请问：
  > 1. 是否需要在 Meegle 中将相关 Story 标记为 Blocked？
  > 2. 是否需要升级给架构师或协调第三方技术支持？

## 4. Meegle 报送专项流程设计

针对用户特别关注的“报送 Bug 到 Meegle 对应的项目”需求，设计以下自动化工作流：

### 4.1 数据准备与映射
在执行自动报送前，必须确保 `project_context.json` 中维护了以下映射关系：
- **Lark User ID ↔ Meegle User Key**：用于将群聊中的提出者（Reporter）和被@的责任人（Assignee）映射到 Meegle 账户。
- **Lark 群组/模块 ↔ Meegle Project ID**：例如，“上线前前端优化需求”群组映射到 Meegle 的 `Frontend_Optimization` 项目。

### 4.2 报送工作流 (The Meegle Pipeline)
1. **意图识别与拦截**：当缓冲区接收到前端群组的消息，LLM 识别其为 `Bug Report`。
2. **完整度校验**：系统检查是否包含 `description`（现象）、`steps`（复现步骤）、`priority`（优先级）。
3. **触发询问（若不完整）**：机器人拦截对话，发送即时询问模板（见 3.1 节）。
4. **用户回复与合并**：用户回复补充信息后，系统将其与原始消息合并，重新计算完整度。
5. **调用调度器 (Dispatcher)**：完整度达标（≥80分）后，状态变更为 `ready`。
6. **API 交互**：调度器调用 Meegle API `POST /api/v1/projects/{project_id}/defects`，Payload 包含：
   - `title`: LLM 提取的 Bug 摘要
   - `description`: 现象 + 复现步骤
   - `priority`: 映射后的优先级
   - `reporter`: 映射后的 Meegle User Key
7. **结果回传**：工单创建成功后，机器人在原群聊中回复：“✅ 已成功在 Meegle 创建 Bug 工单：[DEF-1024] 游戏渲染加载较慢，已指派给 @XXX。”

## 5. 总结与后续建议

通过分析历史数据，AI 机器人在技术前端群聊中具有巨大的介入价值。其核心在于将模糊的、碎片化的技术讨论，通过**结构化的即时询问**，转化为 Meegle 中标准化的可执行工单。

**后续实施建议**：
1. **更新全局规范**：将上述介入场景和 Meegle 报送流程同步至 `.cursor/rules/module2_buffer.md`。
2. **完善提示词**：在 Thread Separation 的 LLM Prompt 中，增加对 `priority` 和 `steps_to_reproduce` 实体提取的权重。
3. **小范围灰度**：建议先在“上线前前端优化需求”群组中开启机器人的即时拦截功能，收集反馈后再推广至全团队。
