# 「周报汇总」触发机制设计文档

## 1. 触发机制设计目标

周报汇总功能需要整合飞书多维表格、Meegle 进度和群聊洞察等多源数据，并调用 LLM 生成综合摘要。由于该过程涉及大量 API 调用和 LLM 推理，耗时较长，且数据具有明显的周期性（每周二至下周二），因此触发机制的设计目标为：

1.  **自动化与定时执行**：减少人工干预，确保每周二自动生成并发布周报。
2.  **按需手动触发**：支持 PM 或团队成员在需要时（如周二下午开会前）手动触发最新周报的生成。
3.  **解耦与异步处理**：避免阻塞主服务（如 Webhook 响应），确保系统稳定性。

## 2. 现有调度能力分析

基于对 `ai-secretary-architecture` 和 `xpbet-frontend-components` 两个仓库的探索，现有系统的调度能力如下：

### 2.1 `ai-secretary-architecture` (后端与跑批层)

*   **`main.py` (FastAPI 服务)**：主要负责接收飞书 Webhook 事件（`im.message.receive_v1`），使用 `BackgroundTasks` 异步处理消息。目前**没有**内置的定时任务调度器（如 APScheduler）。
*   **`run_batch.py` (每日跑批脚本)**：整合了项目进度更新（`daily_progress_updater.py`）、话题追踪和话题收尾等功能。这是一个典型的离线批处理入口，支持通过命令行参数（如 `--dry-run`, `--hours`）控制执行逻辑。
*   **部署环境 (Procfile)**：仅定义了 `web` 进程（`uvicorn main:app`），说明当前系统依赖外部触发器（如 GitHub Actions、Railway Cron 或 Manus 定时任务）来执行离线脚本。

### 2.2 `xpbet-frontend-components/kanban-v2` (前端与数据采集层)

*   **`server/scheduler.ts`**：内置了基于 `setInterval` 的轻量级调度器。
    *   每 6 小时执行一次全量同步（用户、群列表、群成员、多维表格话题）。
    *   每 30 分钟执行一次群消息/话题增量同步。
*   **定位**：该调度器主要用于**高频的数据采集和本地缓存**，不涉及 LLM 推理或复杂的跨源数据聚合。

## 3. 触发方案设计

结合现有架构，周报汇总的触发机制应采用**“外部定时调度 + 离线脚本执行”**的模式，并辅以**“飞书 Bot 指令手动触发”**。

### 3.1 方案一：自动化定时触发（推荐主方案）

由于用户偏好将自动化任务交给 Manus Lite 执行（频率为一天一次），且 `ai-secretary-architecture` 已有成熟的离线跑批模式（`run_batch.py`），周报汇总应作为跑批任务的一部分或独立的定时任务。

**执行链路**：

1.  **调度器**：使用 Manus 的定时任务功能（`schedule` 工具）或 GitHub Actions 的 Cron 触发器。
    *   **频率**：每周二下午 14:00（或根据团队习惯设定的时间）。
2.  **执行入口**：在 `ai-secretary-architecture` 中新增 `run_weekly_report.py` 脚本。
3.  **执行逻辑**：
    *   调用 `fetch_weekly_data.py` 获取飞书多维表格周报。
    *   调用 `meegle_client.py` 获取 Meegle 进度。
    *   调用 `extract_weekly_insights.py` 获取群聊洞察。
    *   调用 LLM 生成综合摘要。
    *   调用 `inject_weekly_updates.py` 将结果写入 `data/dashboard_data.json`。
    *   执行 `git commit` 和 `git push`，触发看板前端的自动更新（基于 `VITE_DATA_URL` 机制）。

### 3.2 方案二：飞书 Bot 指令手动触发（补充方案）

为了满足 PM 随时查看最新周报的需求，可通过飞书群聊向 AI 秘书发送特定指令来触发周报生成。

**执行链路**：

1.  **触发指令**：PM 在飞书群内发送 `@AI秘书 生成本周周报` 或 `@AI秘书 汇总各模块进度`。
2.  **指令解析**：`main.py` 的 Webhook 接收到消息，`handle_message_event` 函数识别出“生成周报”的意图。
3.  **异步执行**：
    *   `main.py` 立即回复：“收到，正在为您汇总本周（W16）各模块进度，预计需要 2-3 分钟，请稍候...”
    *   将周报生成任务（即调用 `run_weekly_report.py` 的核心逻辑）放入 `BackgroundTasks` 中异步执行。
4.  **结果反馈**：
    *   任务完成后，自动将结果写入 `dashboard_data.json` 并推送到 GitHub。
    *   AI 秘书在飞书群内回复：“✅ 本周周报已生成完毕，看板已更新。您可以点击 [看板链接] 查看详情。”

## 4. 方案对比与总结

| 触发方式 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **自动化定时触发** | 完全自动化，无需人工干预；执行时间固定，团队预期明确。 | 无法应对突发的周报查看需求。 | 每周例行的周报生成（如每周二下午）。 |
| **飞书 Bot 手动触发** | 灵活性高，按需生成；交互自然，符合 PM 使用习惯。 | 占用 Webhook 服务的后台资源；频繁触发可能导致 LLM 成本增加。 | 临时会议前的进度确认；周报生成失败时的手动重试。 |

**最终建议**：采用**双轨制触发机制**。以**自动化定时触发**（依赖 Manus 定时任务或 GitHub Actions）为主，确保每周的基线数据更新；以**飞书 Bot 手动触发**为辅，提供灵活的按需生成能力。这种设计既符合现有架构的解耦原则，又能最大化满足用户的使用偏好。
