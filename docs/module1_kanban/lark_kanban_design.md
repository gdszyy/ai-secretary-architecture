# Lark 多维表格看板设计方案

## 1. 设计目标

作为 AI 项目秘书的产品侧功能看板，Lark 多维表格需要提供直观、清晰的项目管理视图，方便产品经理和项目经理掌握各个模块和功能的进度。本设计基于现有的 XPBET 产品架构多维表格（包含“模块表”和“功能表”）进行扩展和优化。

## 2. 数据模型与字段设计

### 2.1 模块表 (Module Table)

模块表用于管理项目的高层级模块，作为功能的聚合层。

| 字段名称 | 字段类型 | 说明 | 现有/新增 |
| :--- | :--- | :--- | :--- |
| 模块名称 | 文本 (Text) | 模块的唯一名称（如：用户系统、财务系统） | 现有 |
| 分类 | 单选 (SingleSelect) | 模块的分类（如：T0基础框架） | 现有 |
| 优先级 | 单选 (SingleSelect) | 模块的优先级（如：P0-1月、P1-3月） | 现有 |
| 状态 | 单选 (SingleSelect) | 模块的整体状态（待规划、规划中、开发中、测试中、已上线、已归档） | 现有 (需调整选项) |
| 负责人 | 人员 (User) | 模块的整体负责人 | 现有 |
| 模块说明 | 文本 (Text) | 模块的详细描述 | 现有 |
| 包含功能 | 双向关联 (DuplexLink) | 关联到“功能表”中的具体功能 | 现有 |
| **创建时间** | 创建时间 (CreatedTime) | 记录模块创建的时间 | **新增** |
| **更新时间** | 更新时间 (ModifiedTime) | 记录模块最后更新的时间 | **新增** |
| **备注** | 文本 (Text) | 用于记录 AI 秘书的备忘和特殊说明 | **新增** |

### 2.2 功能表 (Feature Table)

功能表是看板的核心，用于管理具体的开发需求和任务。

| 字段名称 | 字段类型 | 说明 | 现有/新增 |
| :--- | :--- | :--- | :--- |
| 功能名称 | 文本 (Text) | 功能的具体名称（如：登录注册流程） | 现有 |
| 功能说明 | 文本 (Text) | 功能的详细描述 | 现有 |
| 状态 | 单选 (SingleSelect) | 功能的当前状态（待规划、规划中、开发中、测试中、已上线、已归档） | 现有 (需调整选项) |
| 功能优先级 | 单选 (SingleSelect) | 功能的优先级（P0-核心、P1-重要、P2-普通） | 现有 |
| 阶段 | 单选 (SingleSelect) | 所属的开发阶段或里程碑 | 现有 |
| 所属模块 | 双向关联 (DuplexLink) | 关联到“模块表”中的模块 | 现有 |
| 负责人 | 人员 (User) | 功能的具体负责人 | 现有 |
| 文档链接 | 链接 (Url) | 相关的 PRD 或设计文档链接 | 现有 |
| 迭代版本 | 单选 (SingleSelect) | 所属的迭代版本（如：上线版本 V1.0） | 现有 |
| **Meegle ID** | 文本 (Text) | **关联的 Meegle 工作项 ID，用于双向同步** | **新增** |
| **Meegle 链接** | 链接 (Url) | 指向 Meegle 工作项的直接链接 | **新增** |
| **创建时间** | 创建时间 (CreatedTime) | 记录功能创建的时间 | **新增** |
| **更新时间** | 更新时间 (ModifiedTime) | 记录功能最后更新的时间 | **新增** |
| **计划开始** | 日期 (Date) | 计划开始开发的日期 | **新增** |
| **计划完成** | 日期 (Date) | 计划完成开发的日期 | **新增** |

## 3. 视图设计

为了满足不同角色的查看需求，在“功能表”中设计以下三种核心视图：

### 3.1 看板视图 (Kanban View) - 核心工作流

*   **视图名称**: 研发看板
*   **分组字段**: `状态`
*   **卡片显示字段**: `功能名称`、`功能优先级`、`负责人`、`所属模块`、`Meegle ID`
*   **过滤条件**: 排除状态为“已归档”的记录
*   **用途**: 直观展示各个功能所处的开发阶段，方便每日站会和进度追踪。

### 3.2 表格视图 (Grid View) - 详细数据管理

*   **视图名称**: 功能全景表
*   **排序规则**: 按 `所属模块` 升序，再按 `功能优先级` 降序
*   **显示字段**: 显示所有字段
*   **用途**: 用于产品经理进行详细的数据录入、批量修改和全局概览。

### 3.3 甘特视图 (Gantt View) - 进度与排期

*   **视图名称**: 迭代排期甘特图
*   **开始日期字段**: `计划开始`
*   **结束日期字段**: `计划完成`
*   **分组字段**: `迭代版本` 或 `所属模块`
*   **用途**: 用于项目经理把控整体项目进度，识别潜在的延期风险。

## 4. 状态流转设计

功能的状态流转是项目管理的核心。以下是标准的状态流转定义：

1.  **待规划 (Backlog)**: 需求已提出，但尚未进行详细设计和排期。
2.  **规划中 (Planning)**: 产品经理正在撰写 PRD，进行需求评审和技术方案设计。
3.  **开发中 (In Progress)**: 研发团队正在进行编码工作。
4.  **测试中 (Testing)**: 开发完成，已提测，QA 团队正在进行测试。
5.  **已上线 (Done)**: 功能已发布到生产环境。
6.  **已归档 (Archived)**: 需求被取消或长期搁置。

### 4.1 触发条件与权限

*   **待规划 -> 规划中**: 产品经理手动调整，或 AI 秘书接收到“开始规划”指令时自动调整。
*   **规划中 -> 开发中**: **关键节点**。当状态变更为“开发中”时，触发自动化流程，在 Meegle 中创建对应的工作项（如果尚未创建）。
*   **开发中 -> 测试中**: 研发人员在 Meegle 中将工作项状态改为“测试中”时，通过 Webhook/API 自动同步回 Lark。
*   **测试中 -> 已上线**: QA 在 Meegle 中将工作项状态改为“已完成/已上线”时，自动同步回 Lark。

## 5. AI 秘书交互场景

*   **创建需求**: 用户在 Lark 群聊中 @AI秘书 发送 `新增功能：[模块名称] - [功能名称] - [说明]`，AI 秘书自动在 Lark 功能表中创建一条状态为“待规划”的记录。
*   **更新进度**: 用户发送 `[功能名称] 进度更新为开发中`，AI 秘书更新 Lark 状态，并触发向 Meegle 的同步。
*   **查询状态**: 用户发送 `查询 [模块名称] 的进度`，AI 秘书读取 Lark 看板视图，汇总当前各状态的功能数量并回复。
