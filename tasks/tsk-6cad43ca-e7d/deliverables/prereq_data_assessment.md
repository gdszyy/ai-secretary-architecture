# Module1 看板前置数据完备性诊断评估报告

**评估日期**：2026-04-09  
**评估角色**：analyst (Manus AI, agt-2061f2f5-3f9)  
**评估目标**：诊断 Module1（项目看板）正式运行前依赖的前置数据是否完备，并提供优先级排序的数据补全行动清单。  
**数据来源**：`kanban_data.json` (v2)、飞书多维表格导入报告、看板数据结构 Schema v2、看板优化方案

---

## 1. 评估背景与范围

根据《AI Management 看板系统优化与信息源架构设计方案》[1]，看板系统的核心目标是从"被动汇报"升级为"主动感知与智能聚合"。要实现这一目标，看板必须依赖结构化、可追溯的前置数据，形成"大版本迭代 → 模块规划 → 里程碑 → 子任务"的四级追踪体系。

本次诊断评估的核心范围包括以下四个维度：

1. **项目模块清单完整度**：模块名、负责人、状态、epicId 四项字段的覆盖情况
2. **大版本规划（Epic/Release Plan）**：是否存在独立的大版本规划数据对象
3. **里程碑基线（Milestone Baseline）**：里程碑是否建立并具备全局管理能力
4. **团队结构**：模块负责人信息的准确性与完整性

---

## 2. 项目模块清单完整度评估

当前看板数据来源于两个层次：前端展示层（`kanban_data.json`，共 10 个模块）[2] 和底层数据源（飞书多维表格 v2，共 21 个模块）[3]。两层数据的字段完备性评估如下：

| 评估维度 | 评估结果 | 前端层（10模块）| 底层数据源（21模块）|
| :--- | :---: | :--- | :--- |
| **模块名 (Name)** | **已具备** | 10/10 模块均有明确名称 | 21/21 模块均有名称 |
| **状态 (Status)** | **已具备** | 10/10 模块均有 `statusColor` + `statusLabel` | 21/21 模块均有状态枚举值 |
| **负责人 (Owner)** | **部分具备** | 6/10 有研发负责人，8/10 有产品负责人 | 因 API 限制，21个模块的负责人字段均未自动填充 |
| **大版本关联 (epicId)** | **部分具备** | 10/10 模块各有 1 条示例条目含 epicId，整体覆盖率仅 27% | 底层数据无 epicId 字段 |

**详细说明：**

**模块名与状态**已具备完整数据，是看板正常运行的基础保障。前端展示层的状态字段（`statusColor`、`statusLabel`）与底层多维表格的状态枚举值（开发中/规划中/待规划）均已完整覆盖所有模块。

**负责人**字段存在明显断层。前端层中，4 个模块缺失研发负责人（`owner`），2 个模块缺失产品负责人（`productOwner`）；底层多维表格中，由于飞书 API 的人员字段需要有效用户 ID 才能写入，"模块负责人"字段在自动化建表时被整体跳过，21 个模块的负责人数据均需人工补充 [3]。

**epicId 关联**整体覆盖率极低。根据数据统计，前端看板 37 个任务条目中，仅 10 条（每模块各 1 条示例数据）填充了 `epicId`，整体覆盖率仅为 **27.0%**，且模块层级本身并未直接关联大版本。

**结论**：项目模块清单的骨架已建立，但在负责人分配和大版本关联上存在明显断层，无法支撑看板的智能推演与风险预警功能。

## 3. 大版本规划 (Epic/Release Plan) 评估

**评估结果：缺失**

当前系统中缺乏独立的大版本规划数据对象。`epicId` 仅作为看板条目（Items）的一个字符串标签存在（如 `v2.0`、`v3.0-alpha`）[4]，并没有一个集中的地方定义这些版本的具体含义、时间范围和总体目标。历史版本规划文档（`version_planning.md`）虽然记录了 v2025.05 和 v2025.06 两个版本的功能清单，但该文档是一次性的静态规划，并未形成可持续维护的结构化数据对象，也未与模块层级建立强关联。

这种扁平化的标签设计无法支持自上而下的推演和预警，即当底层子任务出现阻塞时，系统无法自动向上标记对应大版本的延期风险。

### 3.1 大版本规划模板设计

为了建立规范的大版本规划体系，建议引入以下独立的数据结构（可在飞书多维表格中新建"版本规划表"，或在 `kanban_data.json` 的 `meta` 层级中新增 `epics` 数组）：

| 字段名称 | 字段类型 | 必填 | 说明 | 示例 |
| :--- | :--- | :---: | :--- | :--- |
| **版本号 (epicId)** | 字符串 | ✅ | 版本的唯一标识符，与 items 中的 epicId 强关联 | `v2.0` |
| **版本名称 (Title)** | 字符串 | ✅ | 版本的简短名称 | `XPBET 全球站基础框架上线` |
| **版本目标 (Objective)** | 文本 | ✅ | 该版本的核心商业或产品目标 | 完成 T0 基础框架全部模块上线，实现基础业务闭环 |
| **开始时间 (Start Date)** | 日期 | ✅ | 版本的计划启动日期 | `2026-01-01` |
| **结束时间 (End Date)** | 日期 | ✅ | 版本的计划发布日期 | `2026-05-30` |
| **状态 (Status)** | 枚举 | ✅ | 规划中 / 进行中 / 已发布 / 已延期 | `进行中` |
| **关联模块 (Modules)** | 关联数组 | ✅ | 该版本涉及的核心模块 ID 列表 | `["sport", "user", "finance", "permission"]` |
| **关键里程碑 (Milestones)** | 关联数组 | 建议 | 该版本下的关键里程碑 ID 列表 | `["ms-001", "ms-002"]` |
| **备注 (Notes)** | 文本 | 可选 | 版本的特殊说明或风险提示 | 依赖 Sportrader 生产 Key 到位 |

**示例数据**（基于现有 epicId 推断）：

| epicId | 版本名称 | 目标 | 时间范围 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| `v2.0` | 全球站基础框架上线 | 完成 T0 基础框架与核心营销模块 | 2026-01 ~ 2026-05 | 进行中 |
| `v3.0-alpha` | 内容中心与代理体系 | 内容管理完整上线，代理体系建立 | 2026-06+ | 规划中 |

## 4. 里程碑基线 (Milestone Baseline) 评估

**评估结果：部分具备**

当前 `kanban_data.json` 中，每个模块包含一个 `milestone` 对象（包含 `date`、`content`、`note` 字段）[2]。这意味着里程碑的概念已经存在，但存在以下三个核心问题：

**问题一：里程碑孤立，缺乏全局视角。** 里程碑嵌套在各模块内部，无法跨模块汇总和追踪，不支持全局里程碑看板视图。

**问题二：截止日期缺失率高。** 在 10 个前端展示模块中，有 3 个模块（数据源灾备、用户系统、内容管理）的里程碑截止日期（`date`）为 `null`，占比 30%。

**问题三：未关联大版本。** 现有里程碑对象缺少 `epicId` 字段，无法实现"大版本 → 里程碑 → 任务"的自上而下追踪链条。

### 4.1 里程碑基线模板设计

为了实现里程碑的动态推演和风险预警，建议将里程碑抽象为独立的数据实体（可在飞书多维表格中新建"里程碑表"，或在 `kanban_data.json` 中新增顶层 `milestones` 数组）：

| 字段名称 | 字段类型 | 必填 | 说明 | 示例 |
| :--- | :--- | :---: | :--- | :--- |
| **里程碑 ID (ID)** | 字符串 | ✅ | 里程碑的唯一标识符 | `ms-sport-001` |
| **里程碑名称 (Name)** | 字符串 | ✅ | 里程碑的简短描述 | `UOF 生产 Key 验证通过` |
| **所属模块 (Module ID)** | 关联 | ✅ | 关联的模块 ID | `sport` |
| **所属大版本 (Epic ID)** | 关联 | ✅ | 关联的大版本 ID，与大版本规划表强关联 | `v2.0` |
| **截止日期 (Due Date)** | 日期 | ✅ | 计划完成的基线日期 | `2026-04-17` |
| **当前状态 (Status)** | 枚举 | ✅ | 正常 / 延期风险 / 已完成 / 已取消 | `正常` |
| **关键交付物 (Deliverables)** | 文本 | ✅ | 验收该里程碑的标准 | 体育注单核心链路全面就绪 |
| **风险说明 (Risk Notes)** | 文本 | 可选 | 当前阻塞或风险描述 | 等待 Sportrader 生产 Key |

## 5. 团队结构与负责人评估

基于 `kanban_data.json` [2] 和多维表格导入报告 [3] 的综合分析，当前团队结构和模块负责人存在以下问题：

### 5.1 缺少负责人的模块清单（前端展示层）

| 模块名 | 模块 ID | 缺失字段 | 当前状态 | 紧急程度 |
| :--- | :--- | :--- | :--- | :--- |
| 数据源灾备 | `datasource` | 研发负责人（`owner`）| 存在阻塞（红色）| **高** |
| 内容管理 | `content` | 研发负责人（`owner`）| 规划中 | 中 |
| 系统性能迭代 | `sysperf` | 研发负责人（`owner`）+ 产品负责人（`productOwner`）| 基本完成 | 低 |
| 用户系统 | `user` | 产品负责人（`productOwner`）| 进行中 | 中 |
| 权限系统 | `permission` | 产品负责人（`productOwner`）| 联调中 | 中 |

### 5.2 底层数据源负责人缺失（飞书多维表格层）

多维表格 v2 的导入报告明确指出，"模块负责人"（21 个模块）和"功能负责人"（114 个功能）字段在自动化建表时因 API 限制被整体跳过，需要人工在飞书界面中进行补充 [3]。这意味着底层真实数据源在责任人分配上存在**系统性缺失**，影响范围为 100%。
