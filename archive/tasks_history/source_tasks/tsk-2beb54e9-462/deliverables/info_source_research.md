## 1. 调研背景与目标

当前 XPBET AI Management 项目看板（Kanban）的数据主要依赖人工录入，缺乏自动化信息采集能力。为解决信息断层、自动化收集能力不足及多任务混杂等痛点，在已明确的“会议记录（音视频转录）”和“Lark 群聊监控（机器人实时捕获）”之外，系统性调研其他可行的信息来源渠道，并对其进行深度分析与优先级排序。

核心目标是：**减少人工录入，实现看板数据的自动化更新，构建端到端的信息源追踪体系**。

## 3. 来源渠道深度分析

### 3.1 GitHub Issues/PRs 与代码仓库活动
**信息类型与价值**：提供最真实的代码级进度反馈。PR 的提交和合并代表了功能的实际落地；Issue 的状态变更反映了 Bug 的修复情况。这能为看板中的“开发中”到“已完成”状态流转提供事实依据。
**接入技术方案**：配置 GitHub Webhook 监听 `pull_request` 和 `issues` 事件，当特定分支（如 main/release）发生合并或 Issue 状态变更时，将 Payload 推送至信息缓冲区。
**挑战与风险**：开发者可能不规范地填写 PR 描述或关联 Issue，导致 AI 难以准确提取模块归属。

### 3.2 Meegle 工作项状态变更
**信息类型与价值**：Meegle 是研发侧的执行引擎。其 Story、Task、Defect 的状态流转、工时消耗、评论更新，是项目管理最结构化的数据源。能直接驱动看板里程碑的推演和延期预警。
**接入技术方案**：通过 Meegle Webhook 监听工作项（Work Item）的状态变更事件，或通过轮询 API 获取增量更新。
**挑战与风险**：Lark User ID 与 Meegle User Key 的身份映射问题（Identity Mapping），需维护全局映射表。

### 3.3 CI/CD 流水线 (GitHub Actions)
**信息类型与价值**：提供“已发布”和“构建失败”的绝对事实。当大版本或模块部署到测试/生产环境时，流水线状态是看板中“测试中”或“已上线”状态的最佳触发器。
**接入技术方案**：在 GitHub Actions 的 YAML 配置中增加步骤，通过 `curl` 向信息缓冲区发送带有执行结果和 Commit Hash 的 Webhook 请求。
**挑战与风险**：告警风暴。频繁的构建失败可能产生大量冗余信息，需在缓冲区进行防堆积和去重处理。

### 3.4 Sentry / 生产监控告警
**信息类型与价值**：提供系统运行时的实时健康度。严重的生产告警（如支付路由失效）可以直接转化为高优先级的缺陷报告（Bug Report）进入看板，并触发紧急通知。
**接入技术方案**：配置 Sentry 的 Webhook 插件，将符合特定规则（如 Error 级别、影响用户数大于阈值）的告警推送到信息缓冲区。
**挑战与风险**：错误日志缺乏业务上下文，AI 难以自动将其归属到具体的业务模块，可能需要提取堆栈信息进行辅助判断。

### 3.5 Lark 审批流与结构化表单
**信息类型与价值**：结构化的需求提报、请假审批（影响可用资源）和发布审批。这些信息天然具备高完整度，无需 AI 进行复杂的实体提取。
**接入技术方案**：利用 Lark 开放平台的“审批事件订阅”和“多维表格表单提交触发器”。
**挑战与风险**：不同审批流的字段结构差异较大，需要针对每种审批表单编写特定的解析映射逻辑。

### 3.6 Telegram 专用汇报群 / Bot
**信息类型与价值**：PM 或研发人员在移动场景下的碎片化输入（如“备忘：明天下午提醒我跟进EPAY路由”）。这是隐性知识和临时待办的重要来源。
**接入技术方案**：开发 Telegram Bot，监听特定群组或私聊消息，调用 `/buffer/items` 接口将其推入信息缓冲区。
**挑战与风险**：信息完整度通常极低，高度依赖缓冲区的 AI 意图识别和多轮补充询问机制（`asking` 状态）。

### 3.7 Lark Wiki / 文档更新
**信息类型与价值**：PRD（产品需求文档）或架构图的重大修改。文档的更新往往意味着需求的变更或技术方案的调整，需要同步反映在看板的子任务中。
**接入技术方案**：订阅 Lark 开放平台的“文档被修改”事件，提取修改摘要。
**挑战与风险**：高频保存会导致大量事件触发，需通过时间窗口或 AI 摘要判断是否为“实质性业务变更”。

### 3.8 日历事件 (Lark Calendar)
**信息类型与价值**：项目里程碑节点、发布窗口期和重要会议。能为看板的里程碑提供时间轴校准。
**接入技术方案**：通过 Lark 日历 API 订阅特定项目日历的变更事件。
**挑战与风险**：日历事件通常只包含标题和时间，缺乏具体的执行状态，主要用于只读展示和提醒。

## 4. 价值/成本比值优先级排序

基于 XPBET 的现有技术栈（Lark, Meegle, GitHub），按照「价值/实施成本」比值进行优先级排序：

1. **Meegle 工作项状态** (价值：极高 | 成本：低 | 比值：最高) - 研发事实的唯一来源。
2. **GitHub Issues/PRs** (价值：高 | 成本：低 | 比值：高) - 代码落地的绝对证据。
3. **CI/CD 流水线 (GitHub Actions)** (价值：高 | 成本：低 | 比值：高) - 交付状态的自动化触发器。
4. **Telegram 专用汇报群/Bot** (价值：高 | 成本：中 | 比值：较高) - 解决移动端碎片信息捕获。
5. **Lark 审批流/表单** (价值：中 | 成本：中 | 比值：中) - 结构化业务流程的接入。
6. **Sentry / 监控告警** (价值：高 | 成本：高 | 比值：中) - 需解决业务上下文映射问题。
7. **Lark Wiki / 文档更新** (价值：中 | 成本：高 | 比值：较低) - 噪音大，需复杂的变更识别。
8. **日历事件** (价值：低 | 成本：中 | 比值：低) - 仅提供时间锚点。

## 5. 调研结论与推荐建议

经过系统性调研，为实现 XPBET AI Management 看板数据的自动化更新，推荐优先接入以下 **4 种**信息来源渠道，作为现有会议记录和群聊监控的强力补充：

1. **Meegle 工作项流转 (首选)**：Meegle 是 XPBET 研发执行的核心引擎。通过 Webhook 接入其状态变更，能以极低的成本获取最准确的任务进度、缺陷修复情况，直接解决看板“执行层数据断层”的痛点。
2. **GitHub PRs 与 CI/CD 状态**：代码合并与流水线部署成功是“任务已完成/已上线”的物理证明。将其接入缓冲区，能彻底消除研发人员“改了代码忘记改看板状态”的顽疾，实现状态的自动闭环。
3. **Telegram 碎片信息 Bot**：作为对 Lark 群聊监控的补充，Telegram Bot 专门处理 PM 在移动端或非正式场合的碎片化指令（如备忘、口头需求）。这充分利用了信息缓冲区的 AI 补全和询问能力，极大提升了项目经理的便利性。
4. **Sentry 生产告警接入**：将生产环境的严重告警转化为高优先级的缺陷（Defect）推送到看板，打破了运维与项目管理之间的信息壁垒，使得紧急问题能第一时间进入看板的管理视野。

通过构建“Lark群聊/Telegram（沟通层） + 会议转录（决策层） + Meegle（执行层） + GitHub/Sentry（物理层）”的立体信息源矩阵，AI 秘书将具备全方位的项目感知能力，真正实现看板的自动化、智能化运转。
