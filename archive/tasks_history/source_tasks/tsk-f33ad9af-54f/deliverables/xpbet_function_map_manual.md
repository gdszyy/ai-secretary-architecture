## 1. 体系概述

XPBET 全球站功能地图管理系统采用「三层联动」架构，旨在为产品、研发、运营及管理层提供从宏观到微观的无缝协作体验。本手册面向业务人员，详细说明如何在三层体系中进行日常操作与协作。

### 1.1 三层架构说明

*   **第一层（全局查看）：自研前端可视化页面**
    *   **定位**：提供全局视角的业务大图，支持快速了解系统全貌。
    *   **技术**：基于 React + AntV G6 构建的思维导图。
    *   **用途**：管理层汇报、跨部门对齐、新人快速了解系统。
*   **第二层（调整细分功能）：飞书多维表格**
    *   **定位**：核心数据管理与状态流转中枢。
    *   **结构**：包含“模块表”（21个模块）和“功能表”（114个功能点）。
    *   **用途**：产品经理规划需求、项目经理追踪进度、研发更新状态。
*   **第三层（新增细节执行）：飞书云文档**
    *   **定位**：详细的需求说明与技术方案沉淀。
    *   **内容**：PRD（产品需求文档）、技术架构图、API 设计等。
    *   **用途**：研发人员查看详细需求、QA 编写测试用例。

## 2. 飞书多维表格字段规范

飞书多维表格是整个体系的数据核心。为了保证数据的准确性和一致性，所有业务人员必须严格遵守以下字段填写规范。

### 2.1 模块表 (Module Table)

模块表用于管理高层级的业务模块。

| 字段名称 | 字段类型 | 必填/选填 | 填写规范与说明 |
| :--- | :--- | :--- | :--- |
| **模块名称** | 文本 | 必填 | 模块的唯一标识，需简明扼要（如：用户系统、财务系统）。 |
| **分类** | 单选 | 必填 | 必须从下拉列表中选择：`T0基础框架`、`T1营销框架-获客`、`T1营销框架-运营`、`T1营销框架-数据分析`、`T2商户管理`。 |
| **优先级** | 单选 | 必填 | 模块的整体开发优先级：`P0-1月`、`P1-3月`、`P2-6月`。 |
| **状态** | 单选 | 必填 | 模块当前整体状态：`待规划`、`规划中`、`开发中`。 |
| **负责人** | 人员 | 必填 | 模块的整体负责人（通常为产品总监或核心 PM）。 |
| **模块说明** | 文本 | 选填 | 简要描述该模块包含的核心业务价值。 |
| **包含功能** | 双向关联 | 自动 | 关联到“功能表”中的具体功能，由系统自动维护，无需手动填写。 |

### 2.2 功能表 (Feature Table)

功能表用于管理具体的开发需求和任务。

| 字段名称 | 字段类型 | 必填/选填 | 填写规范与说明 |
| :--- | :--- | :--- | :--- |
| **功能名称** | 文本 | 必填 | 具体的业务功能点名称（如：登录注册流程）。 |
| **所属模块** | 双向关联 | 必填 | 必须关联到“模块表”中的一个具体模块。 |
| **状态** | 单选 | 必填 | 当前所处阶段：`待规划`、`规划中`、`待需求评审`、`待技术评审`、`开发中`、`测试中`、`完成`。 |
| **功能优先级** | 单选 | 必填 | 功能的优先级：`P0-核心`、`P1-重要`、`P2-一般`、`P3-次要`。 |
| **阶段** | 单选 | 必填 | 所属里程碑：`1月SR验证`、`3月基础上线`、`6月后续迭代`、`6月迭代`。 |
| **迭代版本** | 单选 | 必填 | 所属 Sprint：`Sprint 0`、`Sprint 1 (1.22-2.14)`、`Sprint 2 (2.24-3.9)`、`Sprint 3 (3.10-3.23)`、`上线版本 V1.0`、`待规划`。 |
| **负责人** | 人员 | 必填 | 该功能的具体执行负责人（PM 或研发 Owner）。 |
| **文档链接** | 超链接 | 选填 | **关键字段**：关联到飞书云文档的 PRD 链接（详见第 3 节）。 |
| **功能说明** | 文本 | 选填 | 功能的简要描述或备注。 |
| **简化方案** | 文本 | 选填 | 针对复杂功能的 MVP（最小可行性产品）简化实现方案。 |
| **前置资源** | 文本 | 选填 | 该功能开发所依赖的前置条件（如：需第三方 API 密钥）。 |

## 3. 多维表格与云文档的双向关联方案

为了确保宏观进度与微观细节的无缝衔接，必须规范化管理功能点的「文档链接」字段。

### 3.1 从多维表格到云文档（正向关联）

1.  **创建文档**：产品经理（或 AI 秘书）应优先使用 Markdown 导入方案创建 PRD 文档。先在本地编写标准 Markdown 文件，然后通过 `lark-md-import` 技能（或执行 `import_md.py` 脚本）将其一键导入为飞书云文档。
2.  **获取链接**：导入成功后，脚本会返回文档的 URL 链接；如果是手动创建，点击文档右上角的“分享”复制链接。
3.  **填入表格**：回到飞书多维表格的“功能表”，找到对应功能，将链接粘贴到 **「文档链接」** 字段中。
4.  **规范要求**：
    *   必须使用飞书云文档的官方链接（`https://*.larksuite.com/docx/...` 或 `.../wiki/...`）。
    *   禁止填入非飞书体系的外部链接，以保证权限的统一管理。
    *   文档创建请遵循 [飞书云文档创建规范](../../docs/lark-doc-creation-spec.md)。

### 3.2 从云文档到多维表格（反向关联）

为了方便在阅读 PRD 时快速查看其在整体项目中的状态，建议在 PRD 文档的头部添加反向关联。

1.  **复制记录链接**：在飞书多维表格中，右键点击该功能所在行的行号，选择“复制记录链接”。
2.  **嵌入文档**：在 PRD 文档的“基本信息”表格中，添加一行“所属功能地图”，将复制的记录链接粘贴进去。飞书会自动将其解析为一张信息卡片，实时显示该功能的状态和优先级。

## 4. 自动化流程设计（状态变更通知）

为了减少沟通成本，系统利用飞书多维表格的自动化功能，配置了状态变更触发器。

### 4.1 触发器配置方案

*   **触发条件**：当“功能表”中的 **「状态」** 字段发生变更时。
*   **执行动作**：发送飞书机器人消息到指定的项目群（如：XPBET 研发核心群）。
*   **消息模板**：
    ```text
    🔔 【状态更新通知】
    功能名称：{{功能名称}}
    所属模块：{{所属模块}}
    当前状态：由 {{变更前状态}} 变更为 {{状态}}
    负责人：{{负责人}}
    迭代版本：{{迭代版本}}
    文档链接：{{文档链接}}
    ```

### 4.2 核心流转节点通知

重点关注以下几个关键状态的流转：
1.  **规划中 -> 待需求评审**：通知研发和 QA 准备参与评审。
2.  **待技术评审 -> 开发中**：标志着正式进入研发阶段，同步触发 Meegle 工作项的创建（由 AI 秘书后台处理）。
3.  **开发中 -> 测试中**：通知 QA 团队开始介入测试。
4.  **测试中 -> 完成**：发送捷报，通知全员该功能已可交付。

## 5. 前端页面与飞书工作台的集成方案

为了让业务人员无需离开飞书即可查看全局可视化页面，我们将自研前端页面集成到飞书工作台中。

### 5.1 集成方式：飞书网页应用 (Web App)

考虑到前端页面（React + AntV G6）的交互复杂性和渲染面积需求，采用**飞书网页应用**的方式进行集成，而非空间受限的小组件（Gadget）。

### 5.2 配置步骤（管理员操作）

1.  **部署前端**：将自研前端页面部署到公司内部服务器或云平台，获取访问 URL（如 `https://xpbet-map.internal.com`）。
2.  **创建应用**：在飞书开发者后台创建一个企业自建应用。
3.  **配置网页**：在应用的“网页”功能中，启用网页能力，并将上述 URL 填入“桌面端主页”和“移动端主页”。
4.  **发布上线**：配置应用权限并发布版本。
5.  **工作台添加**：业务人员在飞书工作台中搜索该应用名称（如“XPBET 功能地图”），点击添加即可直接在飞书客户端内全屏打开可视化页面。

## 6. 权限管理方案

为了保证数据安全和视图的清晰，针对不同角色设计了不同的视图和权限。

### 6.1 角色与权限矩阵

| 角色 | 多维表格权限 | 云文档权限 | 前端页面权限 |
| :--- | :--- | :--- | :--- |
| **PM (产品经理)** | **可编辑**（所有字段） | **可编辑**（PRD） | 可见 |
| **开发/QA** | **部分编辑**（仅可修改自己负责功能的“状态”） | **可评论**（PRD） | 可见 |
| **运营** | **仅查看** | **仅查看** | 可见 |
| **老板/管理层** | **仅查看** | **仅查看** | 可见 |

### 6.2 多维表格视图设计

在飞书多维表格中，预设了以下视图供不同角色快速切换：

1.  **全景视图 (Grid)**：显示所有字段，供 PM 进行全局管理和批量编辑。
2.  **研发看板 (Kanban)**：按“状态”分组的看板视图，隐藏说明等长文本字段，供研发和项目经理在每日站会时拖拽更新进度。
3.  **我的任务 (Grid - 过滤)**：过滤条件设置为“负责人 = 当前用户”，供开发和 QA 快速查找自己名下的功能点。
4.  **迭代排期 (Gantt)**：按“迭代版本”分组的甘特图视图，供项目经理把控整体进度。

## 7. 常见问题 (FAQ)

**Q: 为什么我在前端可视化页面上看到的数据不是最新的？**
A: 前端页面通过 Webhook 或定时任务与多维表格同步数据。通常有 1-5 分钟的延迟。如果长时间未更新，请联系系统管理员检查同步脚本。

**Q: 我可以随意修改功能的所属模块吗？**
A: 建议由 PM 统一调整。如果必须修改，请确保新的模块在“模块表”中已存在，并在多维表格中重新建立关联。

**Q: 发现某个功能点没有文档链接怎么办？**
A: 请在多维表格中 @ 该功能的负责人（通常是 PM），提醒其补充 PRD 链接。目前系统统计文档链接覆盖率约为 18.4%，需要大家共同努力完善。
