## 1. 设计背景与原则

### 1.1 核心背景

XPBET 全球站功能地图管理系统采用三层架构，本方案专注于第二层的数据结构设计：

| 层级 | 载体 | 定位 |
| :--- | :--- | :--- |
| **第一层（全局查看）** | React + AntV G6 前端可视化 | 渲染4层树状思维导图（根→分类→模块→功能），供管理层汇报和全局把控 |
| **第二层（调整细分功能）** | 飞书多维表格（本方案） | 核心数据管理与状态流转中枢，供 PM、研发、运营日常维护 |
| **第三层（新增细节执行）** | 飞书云文档 | PRD、技术方案等详细文档，通过「文档链接」字段与第二层关联 |

### 1.2 设计原则

本方案完全忽略现有表格的结构约束，从零开始重新设计，遵循以下三大原则：

**原则一：以「前端数据消费需求」为核心反推**。前端需要渲染4层树状结构（根节点固定，分类、模块、功能为动态数据），并需要状态、优先级、负责人、阶段、迭代版本、文档链接等业务字段用于节点样式渲染、Tooltip 展示和点击跳转。表格的每一个字段都应能直接映射到前端 JSON 的某个属性。

**原则二：兼顾「业务人员操作便利性」**。字段数量控制在合理范围内，枚举值清晰易懂，通过双向关联和 Lookup（查找）字段减少人工重复录入。例如，功能的「所属分类」通过查找其关联模块的分类自动填充，无需人工维护。

**原则三：支持「状态流转自动化」**。「功能状态」和「模块状态」字段采用单选类型，枚举值有明确的状态流转顺序，便于在飞书多维表格的自动化功能中配置触发器，实现状态变更时自动发送通知。

## 2. 表格结构设计

为满足前端4层树状结构的渲染需求，同时保持业务操作的便利性，本方案设计了**两张核心表**：**「模块表」** 和 **「功能表」**。

> **设计决策说明**：根节点（"XPBET 全球站"）为前端固定值，无需在表格中维护。「分类」作为模块的属性字段存在，前端通过对模块的「所属分类」字段进行 Group By 来动态生成 Level 1 分类节点，这样可以避免维护单独的分类表，将业务人员的维护入口集中在两张表内，降低操作复杂度。

### 2.1 模块表 (Modules)

**定位**：管理高层级的业务模块，是功能点的容器。对应前端树状结构的 **Level 2 节点（module）**。

| 字段名称 | 飞书字段类型 | 必填 | 默认值 | 枚举值 / 说明 | 前端消费用途 |
| :--- | :--- | :---: | :--- | :--- | :--- |
| **模块名称** | 文本 (Text) | 是 | — | 模块的唯一标识，需简明扼要（如：用户系统、财务系统）。 | 节点显示文本 `label` |
| **所属分类** | 单选 (Single Select) | 是 | — | `T0基础框架` / `T1营销框架-获客` / `T1营销框架-运营` / `T1营销框架-数据分析` / `T2商户管理` | 生成 Level 1 分类节点，决定分类节点颜色 `data.category` |
| **模块状态** | 单选 (Single Select) | 是 | 待规划 | `待规划` / `规划中` / `开发中` / `已上线` | 节点背景色 `style.fill`，Tooltip 展示 `data.status` |
| **模块优先级** | 单选 (Single Select) | 是 | P2-6月 | `P0-1月` / `P1-3月` / `P2-6月` | 节点边框色 `style.stroke`，Tooltip 展示 `data.priority` |
| **模块负责人** | 人员 (Person) | 是 | — | 模块的整体负责人（产品总监或核心 PM）。 | Tooltip 展示 `data.owner` |
| **模块说明** | 文本 (Text) | 否 | — | 简要描述该模块包含的核心业务价值（建议不超过 100 字）。 | Tooltip 展示 `data.description` |
| **包含功能** | 双向关联 (Link) | 自动 | — | 关联到「功能表」，由系统在功能表中设置「所属模块」时自动维护，无需手动填写。 | 构建树状层级关系（`children` 数组） |

**模块状态枚举值说明**：

| :--- | :--- | :--- |

**模块优先级枚举值说明**：

| 枚举值 | 含义 | 颜色映射 |
| :--- | :--- | :--- |
| P0-1月 | 最高优先级，1月内必须完成 | `#F5222D`（红色） |
| P1-3月 | 高优先级，3月内完成 | `#FA8C16`（橙色） |

## 3. 表间关联关系（ER 图文字描述）

本系统采用极简的双表结构，通过双向关联建立层级关系。

```
┌─────────────────────────────────────────┐
│           模块表 (Modules)               │
│─────────────────────────────────────────│
│ PK  Record ID (飞书内置)                 │
│     模块名称 (Text)                      │
│     所属分类 (Single Select)             │
│     模块状态 (Single Select)             │
│     模块优先级 (Single Select)           │
│     模块负责人 (Person)                  │
│     模块说明 (Text)                      │
│     包含功能 (Link → 功能表)  ─────────┐ │
└─────────────────────────────────────────┘ │
                                             │ 1 : N
┌─────────────────────────────────────────┐ │
│           功能表 (Features)              │ │
│─────────────────────────────────────────│ │
│ PK  Record ID (飞书内置)                 │ │
│     功能名称 (Text)                      │ │
│     所属模块 (Link → 模块表)  ◄──────────┘ │
│     所属分类 (Lookup from 所属模块)      │
│     功能状态 (Single Select)             │
│     功能优先级 (Single Select)           │
│     里程碑阶段 (Single Select)           │
│     迭代版本 (Single Select)             │
│     功能负责人 (Person)                  │
│     文档链接 (URL)                       │
│     功能说明 (Text)                      │
│     简化方案 (Text)                      │
│     前置资源 (Text)                      │
└─────────────────────────────────────────┘
```

**关系说明**：

一个「模块」可以包含多个「功能」（1:N 关系）。一个「功能」必须且只能属于一个「模块」。在飞书多维表格中，通过在「功能表」中设置「所属模块」为双向关联字段，系统会自动在「模块表」中生成「包含功能」字段，两者构成双向关联。「功能表」中的「所属分类」字段通过 Lookup（查找）功能，自动读取其「所属模块」的「所属分类」值，保证数据一致性，无需人工重复填写。

## 4. 前端数据接口映射表

前端获取飞书多维表格数据后，需要将其转换为 G6 MindMap 所需的节点数据模型。以下是表格字段到前端 JSON 属性的完整映射关系。

### 4.1 根节点（固定值，无需从表格读取）

```json
{
  "id": "root",
  "label": "XPBET 全球站",
  "type": "root",
  "style": {
    "fill": "#001529",
    "stroke": "#001529",
    "lineWidth": 0
  }
}
```

### 4.2 分类节点（由模块表的「所属分类」字段动态生成）

前端对模块表记录按「所属分类」字段进行 Group By，为每个唯一分类值生成一个分类节点。

| 数据来源 | 前端 JSON 属性路径 | 数据处理逻辑 |
| :--- | :--- | :--- |
| 「所属分类」字段的唯一值 | `id` | 拼接前缀：`cat_${分类名称}` |
| 「所属分类」字段的唯一值 | `label` | 直接赋值 |
| — | `type` | 固定值：`'category'` |
| 「所属分类」字段的唯一值 | `style.fill` | 根据分类颜色映射表转换为 Hex 颜色值 |
| 「所属分类」字段的唯一值 | `data.category` | 直接赋值 |

**分类颜色映射**：

| 所属分类 | `style.fill` | `style.stroke` |
| :--- | :--- | :--- |
| T0基础框架 | `#E6F7FF` | `#91D5FF` |
| T1营销框架-获客 | `#F6FFED` | `#B7EB8F` |
| T1营销框架-运营 | `#FFFBE6` | `#FFE58F` |
| T1营销框架-数据分析 | `#FFF7E6` | `#FFD591` |
| T2商户管理 | `#F9F0FF` | `#D3ADF7` |

### 4.3 模块节点（来自「模块表」）

| 飞书「模块表」字段 | 前端 JSON 属性路径 | 数据处理逻辑 |
| :--- | :--- | :--- |
| Record ID（飞书内置） | `id` | 拼接前缀：`mod_${RecordID}` |
| 模块名称 | `label` | 直接赋值 |
| — | `type` | 固定值：`'module'` |
| 模块状态 | `style.fill` | 根据状态颜色映射表（见 2.1 节）转换为 Hex 颜色值 |

## 5. 飞书多维表格创建步骤清单

供人工按步骤搭建飞书多维表格的完整操作指南。

### 步骤 1：创建全新多维表格

1. 在飞书云文档中，点击「新建」->「多维表格」。
2. 命名为「XPBET 全球站功能地图 (v2)」。
3. 删除默认生成的「数据表 1」空表（右键 -> 删除），准备从零开始创建。

### 步骤 2：创建「模块表」

1. 点击左下角「+ 添加数据表」，命名为「模块表」。
2. 按以下顺序配置字段：

| 序号 | 字段名称 | 飞书字段类型 | 配置说明 |
| :---: | :--- | :--- | :--- |
| 1 | 模块名称 | 文本 (Text) | 将默认的「多行文本」字段重命名为「模块名称」，保持文本类型。 |

### 步骤 3：创建「功能表」

### 步骤 4：配置自动化工作流（状态变更通知）

1. 点击右上角「自动化」->「创建工作流」，命名为「功能状态变更通知」。
2. **触发器**：选择「字段值变更时」，数据表选择「功能表」，监听字段选择「功能状态」。
3. **操作**：选择「发送飞书消息」，选择目标群聊（如研发核心群），配置消息模板：

4. 可针对关键状态流转节点（如「开发中」、「测试中」、「完成」）分别创建独立的自动化工作流，以便发送更有针对性的通知。
5. 启用工作流。

在「功能表」中，按以下方案创建多个视图，满足不同角色的使用需求：

### 步骤 6：配置前端 API 访问权限

1. 登录飞书开发者后台（[open.feishu.cn](https://open.feishu.cn)），创建或使用已有的企业自建应用。
2. 在应用权限管理中，申请以下权限：
   - `bitable:app:readonly`（读取多维表格）
   - `contact:user.base:readonly`（读取用户基本信息，用于显示负责人）
3. 获取新建多维表格的 `BASE_ID`（在多维表格 URL 中，格式为 `https://xxx.feishu.cn/base/{BASE_ID}`）。
4. 获取「模块表」和「功能表」的 `TABLE_ID`（在多维表格设置中查看，或通过 API 枚举）。
5. 将 `BASE_ID`、`模块表 TABLE_ID`、`功能表 TABLE_ID` 更新到前端代码的配置文件中。

## 6. 数据迁移方案

如需将现有飞书多维表格（BASE_ID=CyDxbUQGGa3N2NsVanMjqdjxp6e）中的数据迁移到新表格，可按以下步骤操作：

1. **导出现有数据**：在现有表格中，选择「导出」->「CSV 文件」，分别导出「模块表」和「功能表」。
2. **清洗数据**：对照新表格的字段定义，检查并清洗枚举值（如将现有「分类」字段的值映射到新「所属分类」字段的枚举值）。
3. **批量导入**：在新表格中，选择「导入」->「CSV 文件」，按字段映射关系导入数据。
4. **重建关联**：CSV 导入后，「所属模块」双向关联字段需要人工重新建立，或通过飞书 API 批量更新。
5. **验证数据**：导入完成后，通过前端可视化页面验证数据是否正确渲染。

## 7. 与现有设计的对比说明

本方案与前序任务（tsk-9103d528-937）中分析的现有表格结构相比，主要改进如下：

| 改进点 | 现有结构 | 新设计方案 | 改进原因 |
| :--- | :--- | :--- | :--- |
| 字段命名 | 「分类」（模块表）、「功能优先级」（功能表） | 「所属分类」、「功能优先级」（统一命名规范） | 提升字段语义清晰度，避免歧义 |
| 分类字段 | 仅存在于模块表，功能表通过公式计算 | 功能表通过 Lookup 自动带出，而非公式 | Lookup 字段更稳定，不依赖公式计算逻辑 |
| 模块状态枚举 | `开发中`、`规划中`、`待规划` | 新增 `已上线` 状态 | 补充完整的生命周期状态 |
| 功能状态枚举 | 7个状态 | 保持7个状态，顺序调整为流转顺序 | 状态枚举值顺序与流转方向一致，便于看板视图排序 |
| Parent items 字段 | 存在但数据均为空 | 移除 | 现有数据中无实际使用，移除可简化表结构 |
| 前置资源字段 | 存在 | 保留 | 对前端有消费价值（Tooltip 展示） |
