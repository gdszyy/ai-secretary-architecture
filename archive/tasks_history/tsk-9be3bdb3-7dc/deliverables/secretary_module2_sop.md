# AI秘书标准作业程序 (SOP) - 核心模块2：信息缓冲区机制

本SOP是对现有《AI秘书标准作业程序》的扩展，专门针对“信息缓冲区”机制进行说明。该机制旨在处理项目经理日常产生的大量碎片化信息，通过AI进行暂存、清洗、补全和分发。

## 1. 机制概述

信息缓冲区是AI秘书处理非结构化信息的中间层。当您通过Telegram、Lark或语音发送信息时，如果信息不完整或意图不明确，AI秘书不会立即在GitHub或Meegle中创建记录，而是将其放入缓冲区。

**核心流程**:
1.  **接收**: 收集来自各渠道的碎片信息。
2.  **暂存与清洗**: AI识别意图、提取关键实体（如模块、优先级、人员），并进行去重和关联。
3.  **补全询问**: 如果信息完整度不足（如缺少关键字段），AI会根据优先级策略主动向您询问。
4.  **归档/派发**: 当信息完整度达标后，自动推送到相应的下游系统（GitHub Issue、Lark多维表格等）。

## 2. 交互指南：如何与缓冲区协作

为了让缓冲区更高效地工作，请了解以下交互模式：

### 2.1 碎片信息的提交

您可以随时发送任何格式的想法、问题或记录，无需严格遵循原有SOP的格式要求。

**示例**:
*   “刚才开会说支付路由有点问题，记得查一下。” (意图：备忘/Bug，缺失：具体模块、优先级)
*   “用户反馈注册收不到验证码。” (意图：Bug，缺失：影响范围、复现步骤)
*   “下周要把活动中心的预算配置功能加上。” (意图：需求，缺失：具体细节)

### 2.2 响应AI的补全询问

当AI秘书认为信息不足以采取行动时，会向您发起询问。请根据提示补充信息。

**即时询问 (高优先级)**:
*   **AI**: “[紧急确认] 关于支付系统的Bug反馈。为了尽快推进，请补充：1. 具体的复现步骤是什么？2. 目前影响了哪些用户群体？”
*   **您的回复**: “1. 在iOS端使用微信支付时必现。2. 影响所有新用户。”

**批量询问 (中/低优先级)**:
*   **AI**: “[日常整理] 今天有2条信息需要您确认细节：1. 关于‘活动中心预算配置’，这属于哪个模块？是否需要设置提醒时间？2. 关于‘注册验证码问题’，是否已解决？”
*   **您的回复**: “1. 属于活动中心模块，明天下午提醒我。2. 已解决，不用管了。”

### 2.3 主动管理缓冲区

您可以通过指令主动查询或清理缓冲区中的信息：

*   **查询待办**: `查询缓冲区待补充信息` 或 `我还有哪些信息没确认？`
    *   **AI行为**: 列出所有状态为 `asking` 的信息条目。
*   **强制归档**: `把刚才那条关于支付的记录直接归档，不用补充了。`
    *   **AI行为**: 将该条目标记为 `ready` 并推送到下游系统（可能信息不全，但强制执行）。
*   **废弃信息**: `取消上一条备忘。` 或 `那条关于验证码的记录作废。`
    *   **AI行为**: 将该条目标记为 `discarded`，不再跟进。

## 3. 信息清洗规则说明

AI秘书在后台会自动执行以下清洗规则，以减少您的负担：

1.  **去重 (Deduplication)**: 如果您在短时间内发送了多条相似的信息（例如，在不同群里重复提到了同一个Bug），AI会识别并合并它们，只保留一条记录。
2.  **关联识别 (Correlation)**: 如果新的碎片信息与现有的GitHub Issue或Meegle任务高度相关，AI会尝试将其作为评论追加到现有记录中，而不是创建新记录。
3.  **意图提取 (Intent Extraction)**: AI会自动判断信息是Bug报告、需求建议、进度更新还是个人备忘，并据此决定后续的路由策略。
4.  **模块归属判断 (Module Assignment)**: 即使您没有明确说明，AI也会根据上下文和历史记录，尝试推断该信息属于哪个项目模块（如：提到“支付路由”自动归类到“支付系统”）。

## 4. 缓冲区防堆积与健康管理

为了防止缓冲区内积累大量未处理的碎片信息，导致系统响应变慢或重要信息被淹没，AI 秘书会执行自动清理策略，同时也需要您的配合。

### 4.1 自动清理策略

*   **自动降级**: 如果 AI 向您询问补充信息超过 24 小时未获回复，该信息将被降级为“低优先级”，并移入“待认领池”，停止主动打扰。
*   **强制归档**: 在“待认领池”中滞留超过 72 小时的信息，将被标记为废弃，并打包归档到每周的“未处理碎片信息汇总”中，从活跃缓冲区中删除。
*   **批量合并**: 当缓冲区滞留量超过 50 条（危险阈值）时，AI 会强制将相似信息合并，并推送到 GitHub 的“杂项” Issue 中清空缓冲区。

### 4.2 项目经理 (PM) 介入指南

*   **每日清零 (Inbox Zero)**: 建议每天下班前，响应 AI 秘书的“批量询问”，处理完当天的待补充信息。
*   **定期审查待认领池**: 每周五下午，使用指令 `查询待认领池`，快速扫视被自动降级的信息。回复 `恢复 [条目ID]` 补充信息，或回复 `清空待认领池`。
*   **提高初始输入质量**: 尽量在首次输入时提供“模块名称”和“核心意图”，这能极大提高 AI 的解析成功率，直接跳过缓冲区进入工作区。

## 5. 与现有SOP的兼容性

本模块机制完全兼容现有的核心指令格式（如 `[模块名称]：[具体内容]`）。如果您使用了标准格式，信息通常会获得较高的完整度评分，从而跳过“补全询问”阶段，直接进入“归档/派发”流程，实现最高效率。

建议在时间允许的情况下，尽量使用标准格式；在紧急或不方便打字时，可以随意发送碎片信息，由缓冲区机制为您兜底。
